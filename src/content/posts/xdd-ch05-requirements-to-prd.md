이 글은 **[SDD+TDD+DDD] 시리즈**의 다섯 번째 글입니다. <br />

Ch.3과 Ch.4에서는 DDD의 기본 어휘를 두 편에 걸쳐 정리했습니다. `Bounded Context`, `Value Object`, `Entity`, `Aggregate`, `Repository`, `Domain Event` 같은 단어가 각각 무엇을 뜻하는지 먼저 맞췄습니다. 이제 남은 질문은 하나입니다. **실제 요구사항을 받으면, 그 개념들을 어디에 어떻게 적용할 것인가?**

이번 글부터는 작은 도입 예제를 벗어나, 조금 더 현실적인 도메인으로 들어가 보겠습니다.

많은 경우 시간이 조금 비면 별다른 목적 없이 릴스나 쇼츠를 보며 시간을 흘려보내기 쉽습니다. 반면 정말 보고 싶은 긴 글, 기술 아티클, 영상, 팟캐스트는 "나중에 봐야지" 하고 지나치다가 결국 잊어버리기 쉽습니다. 즉 문제는 두 가지입니다. **짧고 즉흥적인 콘텐츠는 쉽게 소비하는데, 정작 의미 있는 콘텐츠는 저장해 두고도 다시 꺼내 보지 못한다는 것**입니다. 그리고 시간이 지나면 내가 무엇을 얼마나 소비했는지도 잘 남지 않습니다.

그래서 이번에 다룰 대상은 이 문제를 해결하려는 **좋은 콘텐츠를 미리 저장해 두고, 나중에 의도적으로 소비하고, 그 기록을 관리하는 개인용 도구**입니다. 평소에 발견한 좋은 콘텐츠를 미리 저장해 두고, 여유가 생겼을 때 숏폼 대신 이런 콘텐츠를 꺼내서 의도적으로 소비합니다. 그리고 거기서 멈추지 않고, 무엇을 얼마나 봤는지 기록하고, 나중에는 소비 통계와 추천까지 제공해 **즉흥적 소비를 줄이고 더 의식적인 소비 습관을 돕는 것**이 목표입니다.

물론 이것은 책에서 설명하기 위해 고른 예제일 뿐입니다. 중요한 것은 독자 여러분도 **자신이 만들고 싶은 서비스나 현재 다루고 있는 도메인으로 바꾸어 생각해 보는 것**입니다. 여기서 보실 것은 특정 서비스 자체보다, 그 서비스를 설계 가능한 형태로 정리해 나가는 절차입니다. 여러분이 만들고 싶은 서비스가 있다면, 같은 방식으로 문제를 적고, 요구사항을 정리하고, 그다음 DDD 관점에서 경계와 객체를 나누어 보시면 됩니다.

다만 곧바로 DDD 용어부터 사용하지는 않겠습니다. 먼저 만들고 싶은 서비스가 무엇인지, 어떤 문제를 풀려는지, 어떤 요구사항이 있는지를 DDD와 무관하게 정리합니다. 이번 글에서는 그 정리의 결과물인 **PRD(Product Requirements Document)** 까지 만드는 것이 목표입니다.

## 먼저 만들려는 서비스를 설명해 보자

이 도구가 풀고 싶은 문제는 단순합니다. 평소에는 좋은 콘텐츠를 발견해도 일단 저장만 해 두고 잊어버리기 쉽습니다. 반대로 시간이 조금 비면 아무 생각 없이 짧은 콘텐츠를 소비하게 되기도 합니다.

그래서 만들고 싶은 것은 이런 흐름을 돕는 개인용 도구입니다.

- 좋은 콘텐츠를 발견하면 바로 저장한다
- 나중에 시간이 날 때 저장해 둔 콘텐츠를 꺼내 본다
- 콘텐츠를 실제로 얼마나 소비했는지 기록한다
- 시간이 지나면 소비 패턴을 다시 돌아볼 수 있다
- 이것들이 쌓이면 일주일에 얼마나 콘텐츠를 소비하는지 통계를 볼 수 있다
- 어떤 분야나 장르를 좋아하는지, 완료하는 데 얼마나 시간이 걸리는지 같은 인사이트를 얻을 수 있다
- 소비한 콘텐츠를 기반으로 유사한 좋은 콘텐츠를 추천받을 수도 있다
- 특정 시각에 너지(nudge) 알림을 보내서, 저장해 둔 콘텐츠를 꺼내 보도록 돕는다

즉 이 서비스의 중심은 "콘텐츠를 발견하는 순간"보다, **저장해 둔 콘텐츠를 나중에 의도적으로 소비하고 관리하는 흐름**에 있습니다.

## AI와 함께 요구사항을 정리한다

서비스의 윤곽이 잡혔으니, 이제 기능 요구사항을 좀 더 구체적으로 정리할 차례입니다. 이 단계에서도 AI가 좋은 대화 상대가 됩니다. 머릿속에 흩어져 있는 생각을 AI에게 던지면, AI가 빠진 부분을 되묻고, 애매한 부분을 구체적으로 만들어 줍니다.

> **나:** 좋은 콘텐츠를 저장하고 나중에 의도적으로 소비하는 도구를 만들고 싶어. 핵심 요구사항을 같이 정리해 보자.

> **AI:** 몇 가지 확인하겠습니다.
>
> 1. 콘텐츠를 저장할 때 어떤 정보가 붙나요? URL만 있으면 되나요, 아니면 태그나 미디어 유형 같은 분류도 필요한가요?
> 2. 저장한 콘텐츠의 상태를 관리하나요? 예를 들면 "아직 안 봤다 / 보는 중 / 다 봤다" 같은 구분이요.
> 3. 다 본 콘텐츠에 대해 별점이나 메모를 남기는 기능이 필요한가요?
> 4. 소비 기록이 쌓이면 어떤 가치를 기대하나요? 통계, 추천 같은 기능도 범위에 들어가나요?

> **나:** URL은 필수고, 태그와 미디어 유형도 붙이고 싶어. 상태 관리가 핵심이야 — 북마크, 보는 중, 다 봤다 정도. 다 본 콘텐츠에는 평가를 남기고 싶고. 그리고 이것들이 쌓이면 일주일에 얼마나 소비했는지 통계를 보고 싶어.

> **AI:** 통계 쪽을 좀 더 구체적으로 잡아 보겠습니다. 어떤 종류의 인사이트를 기대하나요? 예를 들면:
>
> - 주간/월간 소비량 (몇 개 완료했는지)
> - 미디어 유형별 분포 (영상 vs 아티클 vs 팟캐스트)
> - 분야나 장르별 선호도 (어떤 태그의 콘텐츠를 많이 소비하는지)
> - 완료까지 걸린 시간 (북마크 → 완료까지 평균 며칠)
>
> 이런 것들 중 어디까지 필요한가요?

> **나:** 다 좋아. 소비량, 유형별 분포, 장르 선호도, 완료까지 걸린 시간 같은 건 다 보고 싶어. 그리고 소비한 콘텐츠를 기반으로 유사한 좋은 콘텐츠를 추천받을 수도 있으면 좋겠어. 아, 그리고 하나 더 — 특정 시각에 알림을 보내서 저장해 둔 콘텐츠를 꺼내 보도록 너지(nudge)를 주는 기능도 있으면 좋겠어.

> **AI:** 그러면 기능을 네 축으로 나눌 수 있겠습니다.
>
> 1. **북마크 & 상태 관리** — 콘텐츠 저장, 소비 상태 전이, 완료 후 평가
> 2. **소비 통계 & 인사이트** — 주간/월간 소비량, 유형별 분포, 장르 선호도, 완료 소요 시간
> 3. **소비 이력 기반 추천** — 소비 패턴을 바탕으로 다음에 볼 콘텐츠 제안
> 4. **너지 알림** — 설정한 시각에 저장된 콘텐츠를 꺼내 보도록 알림
>
> 한 가지 더 — "볼 계획을 세운" 같은 중간 상태도 필요할까요? 북마크만 해둔 것과, 이번 주에 보겠다고 계획한 것을 구분할 수 있으면 알림이나 통계 쪽에서도 유용할 수 있습니다.

> **나:** 좋은 지적이야. 일단은 단순하게 가되, 상태 전이 순서는 확실히 해 두자.

이렇게 몇 번 주고받으면, 머릿속에 흩어져 있던 요구사항이 빠르게 정리됩니다. 처음에는 "저장하고 나중에 보는 도구" 정도였던 생각이, AI의 되묻기를 거치면서 통계의 구체적인 항목이나 기능의 축까지 잡혔습니다. 중요한 것은 AI가 정리해 준 결과를 그대로 쓰는 것이 아니라, **AI의 되묻기를 통해 자기가 놓치고 있던 부분을 발견하는 과정** 자체입니다.

다만 여기서 곧바로 모든 아이디어를 다 채택할 필요는 없습니다. 예를 들어 `Planned` 같은 중간 상태나 너지 알림은 충분히 흥미로운 기능이지만, 이번 시리즈의 본편에서는 **핵심 흐름을 먼저 드러내기 위해 범위를 조금 줄이겠습니다.** 상태 전이는 `Bookmarked → InProgress → Done` 세 단계로 두고, 너지 알림은 후속 확장 아이디어로 남겨 둡니다.

이 대화를 거치면 이런 요구사항 목록이 나옵니다.

- 좋은 콘텐츠를 발견하면 바로 북마크할 수 있어야 한다
- 북마크한 콘텐츠는 `Bookmarked → InProgress → Done` 상태를 가져야 한다
- 다 본 콘텐츠에는 평가를 남길 수 있어야 한다
- 콘텐츠에는 URL, 미디어 유형, 태그 같은 정보가 붙는다
- 소비 기록이 쌓이면 주간/월간 소비량, 유형별 분포, 장르 선호도, 완료 소요 시간 같은 통계를 볼 수 있어야 한다
- 소비한 콘텐츠를 기반으로 유사한 좋은 콘텐츠를 추천받을 수 있어야 한다
- 너지 알림 같은 기능은 후속 확장 아이디어로 남겨 둘 수 있다

이 단계에서는 아직 `Entity`, `Value Object`, `Aggregate` 같은 말을 하지 않아도 됩니다. 우선은 **서비스가 무엇을 해야 하는지**를 자연어로 붙잡는 것이 먼저입니다.

이 글은 단순히 이 예제 하나를 설명하려는 것이 아닙니다. 독자분들도 각자 만들고 싶은 서비스나 다루고 있는 업무 도메인을 떠올리면서, **같은 순서로 설계를 연습해 보는 가이드**로 읽으면 좋겠습니다. 먼저 서비스가 풀고 싶은 문제를 적고, 그다음 핵심 요구사항을 자연어로 정리한 뒤, 마지막에 그것을 DDD 관점에서 분해해 보는 것입니다.

즉 여기서 보여드릴 과정은 특정 서비스에만 맞는 요령이 아니라, 다른 도메인에도 반복해서 적용할 수 있는 기본 절차입니다. 예제가 달라져도 질문은 비슷합니다.

* "이 서비스의 핵심 흐름은 무엇인가?"
* "어떤 요구사항이 함께 움직이는가?"
* "어디서 경계를 나누어야 하는가?"
* "어떤 값과 주체를 구분해야 하는가?"

차례대로 묻는 방식입니다. 이번 글에서는 앞의 두 질문에 답하며 PRD까지 만들고, 다음 글에서 나머지 두 질문에 답하며 DDD 분해를 진행합니다. 그리고 그 결과를 바탕으로 Ch.7에서는 Spec을 정리하고, Ch.8에서는 그 Spec을 테스트와 코드로 옮길 것입니다.

## 요구사항을 PRD로 정리한다

자연어로 요구사항을 나열하는 것만으로도 좋은 출발점이지만, 이대로 바로 DDD 분해로 들어가면 빠뜨리는 것이 생기기 쉽습니다. 그래서 한 단계를 더 거칩니다. **흩어진 요구사항을 PRD(Product Requirements Document)라는 구조화된 문서로 한 번 정리하는 것**입니다.

PRD는 거창한 문서가 아닙니다. 일반적으로 PRD에는 이런 항목들이 들어갑니다.

| 항목 | 설명 |
|---|---|
| **한 줄 요약** | 이 서비스가 무엇인지 한 문장으로 |
| **문제** | 왜 이 서비스가 필요한가, 어떤 불편을 해결하는가 |
| **핵심 기능** | 이 서비스가 제공하는 주요 기능 목록 |
| **대상 범위** | 어떤 종류의 데이터나 콘텐츠를 다루는가 |
| **비즈니스 규칙** | 기능에 붙는 제약과 규칙 |
| **도메인 개념** | 이 서비스에서 등장하는 핵심 대상과 값 |

이 양식에 따라 콘텐츠 소비 추적 도구의 PRD를 채우면 이렇게 됩니다.

### Watchable PRD

**한 줄 요약**

숏폼 영상을 무의미하게 소비하는 대신, 미리 좋은 콘텐츠를 북마크해두고 여유가 생기면 의도적으로 소비하는 습관을 만드는 개인용 도구.

**문제**

- 시간이 남으면 YouTube Shorts, 릴스 같은 숏폼 영상을 무의미하게 소비하게 된다
- 평소 좋은 콘텐츠를 발견해도 나중에 찾기 어렵다
- 내가 어떤 콘텐츠를 얼마나 소비하는지 파악하기 어렵다

**핵심 기능**

1. **콘텐츠 북마크 & 상태 전이** — 좋은 콘텐츠를 즉시 북마크하고, `Bookmarked → InProgress → Done` 순서로 소비 상태를 관리한다
2. **소비 통계 & 인사이트** — 주간/월간 소비량, 미디어 유형별 분포 등 소비 패턴을 보여준다
3. **소비 이력 기반 추천** — 소비 이력을 바탕으로 다음에 볼 콘텐츠를 제안한다

**대상 범위**

미디어 유형을 가리지 않는다 — 영상, 블로그 글, 기술 아티클, 팟캐스트, 트윗 스레드, 뉴스레터 등.

**비즈니스 규칙**

- 콘텐츠는 `Bookmarked → InProgress → Done` 상태 전이를 가진다
- 완료된 콘텐츠에만 평가(`Rating`)를 부여할 수 있다
- URL은 유효해야 하고 정규화 규칙이 있다
- 태그는 정규화(소문자, 트림)와 길이 제한이 있다

**도메인 개념**

- `Content`: 북마크된 콘텐츠 (식별 + 상태 전이 + 생애주기)
- `ContentUrl`: URL 유효성, 정규화
- `MediaType`: video, article, podcast, tweet, newsletter, other
- `Tag`: 커스텀 태그
- `ConsumptionStatus`: Bookmarked → InProgress → Done
- `Rating`: 소비 완료 후 평가

---

이렇게 한 장의 문서로 정리해 두면, 이후 DDD 분해를 할 때 "어떤 기능이 어떤 컨텍스트에 속하는가", "어떤 규칙이 어떤 객체에 붙는가"를 판단하는 기준점이 됩니다. Spec을 쓸 때도, 테스트를 작성할 때도 이 PRD를 다시 꺼내 볼 수 있습니다.

그리고 이 과정은 AI와 함께하면 훨씬 빠릅니다. 만들고 싶은 서비스를 자연어로 설명하고, "이걸 PRD로 정리해줘"라고 요청하면 AI가 초안을 만들어 줍니다. 그 초안을 읽으면서 빠진 것을 채우고, 애매한 부분을 구체화하는 식으로 몇 번 주고받으면 꽤 쓸만한 PRD가 나옵니다.

여기서 한 가지 분명히 짚고 넘어갈 것이 있습니다. **도메인의 주인은 AI가 아니라 나 자신**입니다. AI는 빠진 항목을 되묻고, 구조를 정리해 주고, 빈틈을 드러내는 데 뛰어납니다. 하지만 "이 서비스가 진짜로 풀어야 할 문제가 무엇인가", "이 기능이 정말 필요한가", "이 규칙이 우리 도메인에 맞는가" 같은 판단은 결국 도메인을 이해하고 있는 사람의 몫입니다.

AI가 만든 초안을 그대로 쓰는 것이 아닙니다. **그 초안을 읽으면서 자기 도메인에 대한 이해를 더 선명하게 만드는 것**이 핵심입니다. AI는 내 생각을 빠르게 구조화해 주는 도구이고, 최종 결정과 책임은 내가 가져가는 것입니다. 이 관계가 뒤집히면 — AI가 제안한 대로 그냥 따라가기 시작하면 — PRD는 그럴듯해 보이지만 실제 도메인과 어긋난 문서가 됩니다.

다시 말해, **어디로 갈 것인지는 내가 정해야 합니다.** 어떤 문제를 풀 것인지, 무엇이 핵심이고 무엇이 부차적인지, 이 서비스가 결국 어떤 가치를 만들어야 하는지는 도메인을 이해하고 있는 사람만이 판단할 수 있습니다. AI는 그 방향이 정해진 뒤에, **거기에 어떻게 도달할 것인지**를 더 빠르고 체계적으로 정리하는 데 도움을 줄 수 있습니다. 빠진 항목을 찾아주고, 구조를 잡아주고, 모호한 부분을 선명하게 만들어 줍니다.

하지만 내가 어디로 가야 할지도 모르고, 무엇을 만들어야 할지도 모른다면, AI가 대신 정해줄 수는 없습니다. AI는 나를 대신할 수 없습니다. 
**내가 방향을 잡고, AI가 속도를 올려 주는 것** — 이것이 이 시리즈에서 AI와 협업하는 기본 원칙입니다.

## PRD에서 구현까지, 전체 흐름을 먼저 보자

지금까지 문제 정의와 PRD 작성을 마쳤습니다. 이제부터 DDD 관점의 설계로 넘어가는데, 그 전에 이 시리즈가 따르는 전체 흐름을 한 번 짚겠습니다.

| 순서 | 단계 | 질문 | 하는 일 | 산출물 | 시점 |
|---|---|---|---|---|---|
| 1 | 문제 정의 | Why | 풀려는 문제와 요구사항을 자연어로 적는다 | 요구사항 목록 | 이번 글 ✅ |
| 2 | PRD 작성 | What | 요구사항을 구조화된 문서로 정리한다 | [`docs/0watchable.PRD.md`](../../docs/0watchable.PRD.md) | 이번 글 ✅ |
| 3 | DDD 전략적 설계 | Where | Bounded Context를 나누고 핵심 컨텍스트를 고른다 | Ch.6의 도메인 분해 결과 | Ch.6 |
| 4 | DDD 전술적 설계 | How | Aggregate → Entity → Value Object를 식별한다 | Ch.6의 객체 후보 목록 | Ch.6 |
| 5 | SDD | Spec | 객체의 규칙을 AI와 함께 Spec으로 명세한다 | `docs/specs/{building-block}/{name}.md` | Ch.7 |
| 6 | TDD | 구현 | Spec을 테스트로 옮기고 Red→Green→Refactor | `tests/` + `src/domain/` | Ch.8 |

이 흐름에서 **PRD (`docs/watchable.PRD.md`)가 모든 후속 작업의 출발점**입니다. DDD 설계도, Spec도, 테스트도 결국 PRD에 적힌 기능과 규칙에서 시작합니다. 그래서 PRD는 한 번 쓰고 끝나는 문서가 아닙니다. DDD 분해를 하다 보면 PRD에서 빠뜨린 규칙이 드러나기도 하고, Spec을 쓰다 보면 객체 경계를 다시 조정해야 할 때도 있습니다. **각 단계의 산출물은 확정본이 아니라 다음 단계의 입력이자, 동시에 피드백을 받아 갱신되는 문서**입니다.

```
docs/
├── 0watchable.PRD.md             ← 요구사항 정리 (DDD 무관)
├── 1xdd-series-guide.DESIGN.md   ← 시리즈 설계/진행 현황/다음 작업
└── specs/
    └── value-object/
        ├── content-url.md        ← 개별 Building Block Spec (Ch.7~)
        ├── tag.md
        └── ...
```

### Tip: 진행 현황 문서로 관리한다

상위 설계를 통해 어떤 Building Block을 만들어야 할지 대략적으로 정리가 되면, 그 목록을 어떤 형태로든 체크박스로 남겨 두는 것이 좋습니다. 전용 DESIGN 문서일 수도 있고, 지금 저장소처럼 시리즈 가이드 문서일 수도 있습니다. 이렇게 하면 프로젝트 전체의 진행 상황을 한눈에 파악할 수 있는 보드가 됩니다. 무엇이 끝났고, 무엇이 남아 있는지가 바로 보이기 때문에, 다음에 작업을 이어 갈 때 **"오늘 어디부터 하면 되지?"** 하고 고민할 필요가 없습니다. 예를 들어 Watchable 전용 메모를 따로 둔다면 이런 형태일 수 있습니다.

```markdown
# Watchable DESIGN

## 진행 현황

### Value Object
- [x] ContentUrl
- [x] Tag
- [ ] MediaType
- [ ] ConsumptionStatus
- [ ] Rating

### Entity
- [ ] Content

### Aggregate
- [ ] Content

---

**한 줄 요약**

숏폼 영상을 무의미하게 소비하는 대신, 미리 좋은 콘텐츠를 북마크해두고 여유가 생기면 의도적으로 소비하는 습관을 만드는 개인용 도구.

**문제**

- 시간이 남으면 YouTube Shorts, 릴스 같은 숏폼 영상을 무의미하게 소비하게 된다
- 평소 좋은 콘텐츠를 발견해도 나중에 찾기 어렵다
- 내가 어떤 콘텐츠를 얼마나 소비하는지 파악하기 어렵다

**핵심 기능**

1. **콘텐츠 북마크 & 상태 전이** — 좋은 콘텐츠를 즉시 북마크하고, `Bookmarked → InProgress → Done` 순서로 소비 상태를 관리한다
2. **소비 통계 & 인사이트** — 주간/월간 소비량, 미디어 유형별 분포 등 소비 패턴을 보여준다
3. **소비 이력 기반 추천** — 소비 이력을 바탕으로 다음에 볼 콘텐츠를 제안한다

**대상 범위**

미디어 유형을 가리지 않는다 — 영상, 블로그 글, 기술 아티클, 팟캐스트, 트윗 스레드, 뉴스레터 등.

**비즈니스 규칙**

- 콘텐츠는 `Bookmarked → InProgress → Done` 상태 전이를 가진다
- 완료된 콘텐츠에만 평가(`Rating`)를 부여할 수 있다
- URL은 유효해야 하고 정규화 규칙이 있다
- 태그는 정규화(소문자, 트림)와 길이 제한이 있다

**도메인 개념**

- `Content`: 북마크된 콘텐츠 (식별 + 상태 전이 + 생애주기)
- `ContentUrl`: URL 유효성, 정규화
- `MediaType`: video, article, podcast, tweet, newsletter, other
- `Tag`: 커스텀 태그
- `ConsumptionStatus`: Bookmarked → InProgress → Done
- `Rating`: 소비 완료 후 평가
```

작업이 끝날 때마다 이 목록을 갱신합니다. TDD까지 마치면 체크박스를 완료로 바꿉니다. 형식이 어떻든 이런 진행 현황 문서가 있으면 **설계 결과 + 작업 현황**을 함께 관리할 수 있습니다.
지금 저장소에서는 이 역할을 `docs/watchable.DESIGN.md` 가 맡고 있습니다. 다음 날 작업을 시작할 때는, PRD와 진행 현황 문서를 AI에게 넘기고 이렇게 묻는 것만으로 바로 이어갈 수 있습니다.

> **나:** `watchable.PRD.md`와 `watchable.DESIGN.md`를 공유할게. 오늘 뭘 해야 하지?

> **AI:** 진행 현황 문서를 보면, `ContentUrl`과 `Tag`는 완료되어 있고, 다음은 `MediaType`입니다. PRD의 비즈니스 규칙을 보면 `MediaType`은 허용 값 집합이 정해져 있는 Value Object이니, 먼저 Spec을 정의하는 것부터 시작하면 되겠습니다.

이처럼 PRD와 진행 현황 문서만 공유하면, AI가 **현재 상태를 파악하고 다음 할 일을 제안**할 수 있습니다. 별도로 맥락을 설명하거나 이전 대화를 찾아볼 필요가 없습니다.
