이 글은 **[SDD+TDD+DDD] 시리즈**의 여덟 번째 글입니다. <br />

Ch.7에서는 `Tag`를 중심으로 Spec을 정의하는 과정을 살펴봤고, `ContentUrl`, `ConsumptionStatus`에도 같은 방식이 적용된다는 점을 봤습니다. 각 객체의 생성 규칙, 불변 조건, 동등성이 문서로 정리되어 있지만, 코드는 아직 한 줄도 없습니다.

이번 글의 질문은 이것입니다. **"Spec이 있으면, 그다음은 뭘 해야 하지?"**

자연스러운 답은 "구현을 시작하자"입니다. 하지만 TDD 흐름에서는 그 전에 한 단계가 더 있습니다. **Spec에서 어떤 테스트를 뽑을지 먼저 정리하는 것**입니다.

이번 글에서는 Ch.7에서 만든 Spec을 기준으로, 무엇을 테스트해야 하는지 정리합니다. 왜 이 단계가 필요한지, 그리고 AI와 함께 개발할 때 어떤 역할을 하는지도 함께 보겠습니다. 실제 테스트 코드 작성과 구현은 다음 글에서 이어집니다.

## TDD란

TDD는 단순합니다. 구현 전에 테스트를 먼저 씁니다.
흐름은 세 단계로 돌아갑니다.

1. **Red**: 아직 구현이 없으니 실패하는 테스트를 쓴다
2. **Green**: 테스트를 통과하는 최소한의 구현을 쓴다
3. **Refactor**: 테스트가 통과하는 상태를 유지하며 코드를 정리한다

이 사이클을 반복하면서 구현을 조금씩 완성해 갑니다.

### 테스트를 먼저 쓰는 진짜 이유

TDD에서 테스트를 먼저 쓰라고 하면, "구현도 안 했는데 뭘 테스트하지?"라는 의문이 자연스럽게 듭니다. 하지만 바로 그 지점이 핵심입니다. 구현 내부로 들어가기 전에, 이 객체를 사용하는 쪽에서 먼저 생각하는 것입니다.

테스트를 먼저 쓰면 자연스럽게 이런 질문부터 하게 됩니다.

- 이 객체를 어떻게 생성하지? `new Tag("ddd")`? `Tag.create("ddd")`?
- 외부에서 어떤 메서드를 호출하지? `.value`? `.equals()`?
- 어떤 입력을 넣으면 어떤 결과가 나와야 하지?

이 질문들은 모두 외부 사용 관점의 질문입니다. 내부에서 어떤 자료 구조를 쓰는지, 검증을 어떤 순서로 처리하는지는 아직 중요하지 않습니다. 먼저 정해야 하는 것은 "이 객체가 밖에서 보기에 어떻게 동작해야 하는가"입니다.

결국 이것은 객체의 공개 인터페이스를 설계하는 일입니다. TDD는 단순히 테스트를 먼저 적는 습관이 아니라, 구현에 끌려가지 않고 인터페이스부터 정하는 방식이라고 볼 수 있습니다.

### 그러면 "무엇을 테스트할지"는 어디서 오는가

여기서 한 가지 질문이 더 남습니다.

> 테스트를 먼저 쓰려면, **무엇을 테스트할지** 알아야 한다.

외부 관점에서 설계한다고 해도, "이 객체가 어떤 규칙을 가져야 하는지"를 모르면 테스트를 쓸 수 없습니다. 빈 파일 앞에서 `it("???")`부터 멈추게 됩니다.

여기서 SDD의 Spec이 바로 연결됩니다. Ch.7에서 정의한 Spec이 있으면 테스트 목록의 뼈대는 이미 나와 있습니다. Spec이 무엇을 검증해야 하는지 알려 주고, TDD는 그 규칙을 실행 가능한 코드로 옮깁니다.

## 왜 TDD가 AI 시대에 강력한가

Ch.7에서는 SDD가 AI 시대에 왜 강력한지를 몇 가지 관점에서 설명했습니다. TDD도 비슷한 방식으로 볼 수 있습니다.

### Spec이 있으면 테스트 목록의 초안이 바로 나온다

Ch.7에서 정리한 `Tag` Spec을 다시 보겠습니다.

```markdown
### 생성 규칙
- 입력값의 앞뒤 공백은 제거한다
- 소문자로 정규화한다
- 정규화한 결과가 비어 있으면 유효하지 않다
- 정규화한 결과가 20자를 초과하면 유효하지 않다
- 내부 공백은 허용한다
```

이 다섯 줄만 봐도 어떤 테스트가 필요한지 윤곽이 잡힙니다.

- 앞뒤 공백을 제거하는가?
- 대문자를 소문자로 정규화하는가?
- 공백만 있는 입력을 막는가?
- 20자를 넘는 태그를 막는가?
- 20자는 허용하는가?
- 내부 공백이 있는 태그를 허용하는가?

Spec 없이 TDD를 시작하면 "뭘 테스트하지?"라는 질문 앞에서 오래 멈추게 됩니다. 어떤 사람은 빈 문자열부터 확인하고, 어떤 사람은 정상 케이스부터 잡습니다. 기준이 없으니 빠뜨리기도 쉽습니다.

반면 Spec이 있으면 출발점이 분명합니다. Spec의 각 줄이 테스트 케이스 후보가 되기 때문입니다.

### 테스트는 Spec이 코드에서 실제로 지켜지는지 감시한다

Spec은 문서입니다. 그리고 문서는 시간이 지나면 코드와 어긋날 수 있습니다. 누군가 `Tag`의 길이 제한을 30자로 바꿨는데, Spec 문서는 여전히 20자라고 적혀 있을 수 있습니다.

테스트는 다릅니다. 테스트는 **실행 가능한 Spec**입니다.

```typescript
it("정규화한 결과가 20자를 초과하면 생성에 실패한다", () => {
  const result = Tag.create("a".repeat(21));
  expect(result.ok).toBe(false);
});
```

이 테스트는 CI에서 매번 실행됩니다. 누군가 길이 제한을 바꾸면, 이 테스트가 실패합니다. 문서는 조용히 낡을 수 있지만, 테스트는 소리를 냅니다.

Spec은 규칙을 적어 두는 문서이고, 테스트는 그 규칙이 실제 코드에서 지켜지는지 확인하는 장치입니다. Spec만 있으면 의도는 남지만 검증이 없고, 테스트만 있으면 검증은 되지만 왜 그런 규칙인지 맥락이 흐려질 수 있습니다. 둘이 함께 있을 때 가장 안정적입니다.

### 테스트는 코드의 사용 설명서다

테스트에는 또 하나의 역할이 있습니다. 구현 코드의 내부를 읽기 전에, 테스트를 먼저 보면 **이 객체를 어떻게 사용하는지**가 바로 드러납니다.

`Tag`의 구현을 처음 보는 사람이 있다고 가정해 봅시다. `src/domain/value-objects/tag.ts`를 열어 `create()` 메서드의 내부 로직을 읽을 수도 있지만, `tests/domain/value-objects/tag.test.ts`를 열면 더 빠르게 파악할 수 있습니다.

```
✓ 앞뒤 공백을 제거한다
✓ 소문자로 정규화한다
✓ 공백만 있는 입력은 생성 실패
✓ 정규화한 결과가 20자를 초과하면 생성에 실패한다
✓ 내부 공백은 허용한다
✓ 같은 값이면 같다
```

이 목록만 읽어도 `Tag`가 어떤 입력을 받고, 무엇을 거부하며, 어떤 규칙으로 동작하는지 파악할 수 있습니다. 구현 코드는 어떻게 동작하는지를 보여주고, 테스트는 무엇을 해야 하고 무엇을 하면 안 되는지를 보여줍니다. 이런 의미에서 테스트를 코드와 함께 유지되는 문서, 즉 living document라고 부르기도 합니다.

이 역할이 제대로 작동하려면 조건이 하나 있습니다. 테스트 이름이 서술적이어야 한다는 점입니다. 이름만 읽어도 어떤 규칙을 검증하는지 알 수 있어야 합니다.

```typescript
// 나쁜 예
it("test1", () => { ... });
it("create error", () => { ... });
```

`"test1"`은 무엇을 검증하는지 전혀 알 수 없습니다. `"create error"`는 조금 낫지만, 어떤 조건에서 어떤 에러인지 드러나지 않습니다. 이런 이름은 테스트가 실패했을 때 코드를 열어봐야 원인을 알 수 있습니다.

```typescript
// 좋은 예
it("앞뒤 공백을 제거한다", () => { ... });
it("정규화한 결과가 20자를 초과하면 생성에 실패한다", () => { ... });
it("Bookmarked에서 start()하면 InProgress가 된다", () => { ... });
```

이름만 읽어도 어떤 입력에서 어떤 결과가 나와야 하는지 파악할 수 있습니다. 테스트가 실패했을 때도 어떤 규칙이 깨졌는지 바로 드러납니다. 테스트 목록을 출력하면 그 자체로 이 객체의 규칙 문서처럼 읽힙니다.

좋은 테스트 이름은 대개 Spec의 한 줄과 비슷한 모양이 됩니다. 실제로 Spec 항목을 테스트 이름으로 옮기면 자연스럽게 서술적인 이름이 나옵니다.

### 테스트 케이스는 어떤 순서로 뽑는가

Spec에서 테스트 목록을 뽑을 때도 어느 정도 순서가 있습니다. 떠오르는 대로 적기보다 몇 가지 원칙을 잡아 두면 빠뜨리기 어렵습니다.

**단순한 것부터 복잡한 것으로.** 가장 기본적인 성공 케이스부터 시작합니다. `Tag.create("ddd")`가 정상적으로 만들어지는지부터 확인하고, 그다음에 공백 제거, 소문자 정규화처럼 변환이 들어가는 케이스로 넘어갑니다. `ConsumptionStatus`라면 단순 생성부터 확인한 뒤, 상태 전이로 복잡도를 높여 갑니다.

**성공하는 경우뿐 아니라 실패하는 경우도.** 테스트는 "이렇게 하면 된다"만 확인하는 것이 아닙니다. "이렇게 하면 안 된다"도 확인해야 합니다. `Tag`라면 빈 문자열, 공백만 있는 문자열, 21자 문자열이 모두 생성에 실패해야 합니다. `ConsumptionStatus`라면 `Bookmarked`에서 `complete()`를 호출하는 것이 실패해야 합니다. 실패 케이스가 없으면, 잘못된 입력이 조용히 통과해도 아무도 모릅니다.

**경계값을 빠뜨리지 않는다.** "20자 이하"라는 규칙이 있으면, 20자(성공)와 21자(실패)를 모두 확인합니다. 경계 바로 위와 바로 아래를 함께 테스트하면, 규칙의 경계가 코드에 정확히 반영되었는지 확인할 수 있습니다.

**중요한 규칙일수록 다양한 케이스를 둔다.** 모든 Spec 항목에 같은 수의 테스트가 필요한 것은 아닙니다. 핵심 비즈니스 규칙이거나 실수하기 쉬운 규칙에는 케이스를 더 두는 편이 좋습니다. 예를 들어 `ConsumptionStatus`의 상태 전이는 허용되는 경우와 허용되지 않는 경우를 모두 빠짐없이 나열하는 편이 안전합니다. 반면 단순한 동등성 검사는 한두 개면 충분합니다.

물론 이 원칙들을 머릿속에 두고 혼자 목록을 뽑을 수도 있습니다. 하지만 AI와 함께하면 더 빠릅니다. Spec 파일을 주고 테스트 케이스를 논의하면, AI는 적힌 항목을 빠뜨리지 않고 정리하고, 사람은 경계값이나 빠진 실패 케이스를 보완합니다. 이 대화를 거쳐 목록을 확정한 뒤 실제 테스트 코드를 쓰는 것이 이 시리즈가 제안하는 순서입니다.

### AI가 코드를 바꿔도, 테스트가 규칙을 지킨다

AI에게 리팩토링을 요청한다고 가정해 봅시다.

> "이 `Tag` Value Object를 더 간결하게 리팩토링해줘."

AI는 구현을 바꿉니다. `create()` 메서드의 내부 로직이 달라질 수 있고, 검증 순서가 바뀔 수도 있습니다. 이때 테스트가 있으면 바뀐 구현이 여전히 규칙을 지키는지 바로 확인할 수 있습니다. 테스트가 모두 통과하면 최소한 핵심 규칙은 유지되고 있다고 볼 수 있습니다.

반대로 테스트가 없다면 AI가 내놓은 코드를 눈으로 한 줄씩 리뷰해야 합니다. "이 구현이 정말 빈 문자열을 막고 있나?", "길이 제한이 여전히 20자인가?" 같은 질문을 사람이 직접 확인해야 합니다. 코드가 많아질수록 놓치기 쉽습니다.

AI에게 구현 자체를 맡기는 경우도 마찬가지입니다.

> "이 Spec대로 `ConsumptionStatus` Value Object를 구현해줘."

AI는 코드를 생성합니다. 이때 테스트가 미리 준비되어 있으면, `pnpm test` 한 번이면 AI의 구현이 Spec을 만족하는지 알 수 있습니다. 테스트가 없으면 AI의 구현을 "믿거나" "직접 하나씩 확인하거나" 둘 중 하나입니다.

이런 의미에서 테스트는 AI가 만든 구현을 자동으로 검증해 주는 리뷰 장치라고 볼 수 있습니다.

### "무엇을 검증할지"가 도메인이다

이 시리즈의 모토는 "도메인에 집중하고, 프레임워크는 AI에게"입니다. TDD에서도 같은 구도가 반복됩니다.
테스트에는 두 가지 층이 있습니다.

| | 예시 | 누가 |
|---|---|---|
| **무엇을 검증할지** (도메인) | "Tag가 빈 문자열을 막아야 한다" | 사람의 판단 |
| **어떻게 검증할지** (기술) | `expect(Tag.create("").ok).toBe(false)` | AI가 잘하는 영역 |

`Tag`가 빈 문자열을 막아야 한다는 것은 도메인 결정입니다. 요구사항을 이해하고, Spec을 정의하고, 그 규칙이 왜 필요한지 아는 사람이 내려야 하는 판단입니다.

반면 그 판단을 `expect(Tag.create("").ok).toBe(false)`라는 코드로 옮기는 일은 기술적 표현입니다. vitest 문법, assertion 패턴, 테스트 구조 같은 것들은 AI가 이미 충분히 잘합니다.

즉 SDD에서 규칙을 정의하는 것이 사람의 역할이었다면, TDD에서는 무엇을 검증할지 정하는 것이 사람의 역할입니다. Spec이 있으면 그 판단은 이미 상당 부분 끝나 있습니다.

> **Spec에서 테스트 목록을 뽑는 것 = 도메인 결정**
>
> **테스트 코드를 작성하는 것 = 기술 결정**

도메인 결정은 사람이 책임지고, 기술적 표현은 AI와 함께합니다. 이것이 이 시리즈에서 말하는 TDD의 역할 분담입니다.

## Spec에서 테스트 목록 뽑기 — AI와의 대화

그러면 실제로 Spec에서 테스트 목록이 어떻게 나오는지 보겠습니다. Ch.7에서 AI와 대화하며 Spec을 정의했듯이, 이번에도 AI와 대화하며 테스트 목록을 뽑아 봅니다. 아직 코드는 쓰지 않습니다. 목록만 정리합니다.

### Tag — 대화

> **나:** `Tag` Spec이 `docs/specs/value-objects/tag.md`에 있어. 이 Spec 기준으로 테스트 케이스 목록을 같이 논의하자. 테스트 이름은 비즈니스 규칙이 드러나도록 서술적으로 적어줘. 구현은 테스트 케이스 목록을 확정한 다음에 할 거야.

> **AI:** `Tag` Spec을 기준으로 테스트 케이스를 정리하겠습니다.
>
> **생성 규칙에서:**
> - 입력값의 앞뒤 공백을 제거한다 — `"  ddd  "` → `"ddd"`
> - 입력값을 소문자로 정규화한다 — `"DDD"` → `"ddd"`
> - 빈 문자열은 생성에 실패한다 — `""`
> - 공백만 있는 입력은 생성에 실패한다 — `"   "`
> - 정규화한 결과가 20자를 초과하면 생성에 실패한다 — 21자 문자열
> - 내부 공백이 있는 태그를 허용한다 — `"machine learning"`
>
> **동등성에서:**
> - 정규화 결과가 같으면 같은 Tag다 — `"DDD"`와 `"ddd"`

> **나:** 20자 초과는 확인하는데, 정확히 20자인 경우는? 경계값도 확인해야 하지 않아?

> **AI:** 맞습니다. 경계값 테스트를 추가하겠습니다.
>
> - 정규화한 결과가 정확히 20자이면 생성에 성공한다
> - 정규화한 결과가 20자를 초과하면 생성에 실패한다
>
> 이렇게 하면 "20자 이하"라는 규칙의 경계가 정확히 어디인지 확인할 수 있습니다.

여기서 중요한 것은, AI가 Spec에서 테스트 목록의 초안을 꽤 잘 뽑아낸다는 점입니다. Spec이 명확하기 때문입니다. 다만 경계값처럼 문서에 직접 적혀 있지 않은 케이스는 사람이 되물어야 합니다. AI는 적힌 항목을 빠뜨리지 않는 데 강하고, 사람은 "이 정도면 충분한가"를 판단하는 데 강합니다.

완성된 Tag 테스트 목록을 정리하면 이렇습니다.

| Spec 항목 | 테스트 케이스 |
|---|---|
| 앞뒤 공백은 제거한다 | `"  ddd  "` → `"ddd"`로 생성된다 |
| 소문자로 정규화한다 | `"DDD"` → `"ddd"`로 생성된다 |
| 비어 있으면 유효하지 않다 | `""` → 생성 실패 |
| | `"   "` → 생성 실패 (trim 후 비어 있음) |
| 20자 초과하면 유효하지 않다 | 21자 문자열 → 생성 실패 |
| | 20자 문자열 → 생성 성공 (경계값) |
| 내부 공백은 허용한다 | `"machine learning"` → 그대로 생성 |
| 두 Tag는 value가 같으면 같다 | `"DDD"`와 `"ddd"` → 같은 Tag |

`ContentUrl`과 `ConsumptionStatus`도 같은 방식으로 진행할 수 있습니다. 패턴은 같습니다. AI가 Spec을 읽고 목록을 만들고, 사람이 빠진 케이스를 보완합니다. Spec이 명확할수록 이 대화는 짧아집니다.

여기서 한 가지 더 볼 점이 있습니다. `Tag`의 테스트 목록과 코드를 먼저 완성해 두면, `ContentUrl`이나 `ConsumptionStatus`를 작업할 때 AI가 이미 만든 `Tag` 파일을 참고할 수 있습니다. 테스트 구조, 이름 컨벤션, assertion 패턴을 앞선 파일에서 읽고 같은 스타일로 맞추기 쉽습니다. 결국 첫 번째 객체에서 패턴을 잘 잡아 두면, 두 번째부터는 그 패턴을 훨씬 일관되게 따라갈 수 있습니다.

AI가 알아서 기존 파일을 찾아보기도 하지만, 명시적으로 알려주면 더 확실합니다.

> "`docs/specs/value-objects/content-url.md` Spec 기준으로 테스트 케이스를 논의하자. `tests/domain/value-objects/tag.test.ts`를 참고해서 같은 패턴으로 작성해줘."

이렇게 하면 AI는 Spec에서 규칙을 읽고, 기존 테스트 파일에서 구조와 스타일을 읽어 둘을 합쳐 새 테스트를 만듭니다.
