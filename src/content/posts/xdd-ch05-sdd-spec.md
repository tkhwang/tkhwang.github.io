이 글은 **[SDD+TDD+DDD] 시리즈**의 다섯 번째 글입니다. <br />

Ch.4에서는 요구사항을 다시 읽으면서, 무엇을 모델링할지 먼저 정했습니다. 그 과정에서 `Customer`는 Entity 후보로, `Name`, `Email`, `RewardPoints`는 Value Object 후보로 분리했습니다. 이번 글의 질문은 하나입니다. **"그래서 그 객체를 어떻게 설계하지?"**

답은 **Spec을 먼저 정의하는 것**입니다. 코드를 작성하기 전에, 이 객체가 어떤 규칙을 가지고 어떤 행동을 하는지를 명세로 정리합니다. 이것이 SDD(Spec-Driven Development)의 핵심입니다. 그리고 이 과정에서 AI가 좋은 대화 상대가 됩니다. 이번 글에서는 AI와 대화하며 Spec을 만들어가는 과정 자체를 시연합니다. Ch.2에서 본 예제를 그대로 이어 가되, 이번에는 실제로 설계에 쓸 수 있을 만큼 규칙을 더 구체화해 보겠습니다.

## SDD란

SDD는 단순합니다. **구현 전에 Spec을 먼저 정의한다.** 그게 전부입니다.<br />
"Spec"이라고 하면 거창하게 들릴 수 있지만, 핵심은 이것입니다.

- 이 객체는 **어떤 조건에서 생성 가능**한가?
- 이 객체는 **어떤 행동**을 하는가?
- 이 객체의 **불변 조건**(invariant)은 무엇인가?

이 질문에 답하는 문서가 곧 Spec입니다. 구현 코드보다 앞서, 도메인 규칙을 먼저 정리하는 것입니다. 그런데 왜 AI와 함께 할까요? 혼자 Spec을 쓰면 내가 이미 알고 있는 것만 적게 됩니다. AI와 대화하면 **내가 미처 생각하지 못한 엣지 케이스**를 짚어줍니다. AI는 "그런데 이 경우는요?"라고 되묻는 상대가 되어 줍니다.

직접 보겠습니다.

## 왜 이 방식이 AI 시대에 강력한가

### AI는 Spec의 빈틈을 먼저 드러낸다

AI가 구현 이전 단계에서 **Spec 자체를 점검하는 상대**가 되어 준다는 것입니다. 사람이 요구사항을 정리할 때는 익숙하다는 이유로 몇 가지를 암묵적으로 넘기기 쉽습니다. "공백만 있는 문자열은?", "0 포인트 적립은?", "정규화는 어디까지?" 같은 질문은 구현을 시작하고 나서야 뒤늦게 떠오르곤 합니다. 하지만 AI와 대화하며 Spec을 쓰면 이런 빈칸이 더 빨리 드러납니다. AI가 예외 상황을 되묻고, 빠진 조건을 제안하고, 모호한 검증 순서를 다시 정리해 주기 때문입니다. 즉 AI는 단순히 Spec을 받아 코드로 옮기는 도구가 아니라, **Spec의 논리적 빈틈을 찾아내고 보강을 돕는 리뷰어** 역할도 할 수 있습니다.

### Spec은 AI에게 주는 명확한 프롬프트다

AI에게 "Name 클래스 만들어줘"라고 하면, AI는 추측합니다. 최대 길이가 255일 수도, 100일 수도 있습니다. 공백을 정리할 수도, 그대로 둘 수도 있습니다.

하지만 위에서 정의한 Name Spec을 AI에게 주면?

> "이 Spec대로 Name Value Object를 TypeScript로 구현해줘."

AI는 추측이 아니라 **명세에 기반한 코드**를 생성합니다. 생성 규칙의 각 항목이 구현의 기준이 되고, 불변 조건은 테스트 케이스를 설계하는 근거가 됩니다.

### Spec은 context 복원 도구다

Ch.1에서 "AI와 장기 프로젝트를 하면 context 유지가 어렵다"고 이야기했습니다. Spec이 이 문제를 줄여줍니다. 일주일 전에 만든 RewardPoints를 수정해야 한다면? RewardPoints Spec 문서를 AI에게 공유하면 됩니다.

> "이 Spec으로 구현된 RewardPoints가 있는데, 여기에 만료일 개념을 추가해야 해. Spec부터 업데이트하자."

Spec이 곧 context입니다. AI의 기억력에 의존하지 않고, 문서화된 명세에 의존합니다. 특히 개발이 진행되면서 Value Object, Entity, Aggregate, Application Service처럼 여러 모듈이 서로 연결되기 시작하면 이 장점이 더 커집니다. 한 객체의 Spec을 바꾸면, 관련된 다른 객체나 유스케이스의 규칙과 충돌하지 않는지도 함께 확인해야 하기 때문입니다.

예를 들어 `ConsumptionStatus`의 상태 전이 규칙을 수정하면, `Content` Entity의 행동, 완료된 콘텐츠에만 허용되는 `Rating` 규칙, 통계 집계 기준까지 함께 살펴봐야 할 수 있습니다. 이때 AI에게 관련 Spec들을 함께 주면, 변경이 어디에 영향을 주는지 빠르게 짚고, 보강이 필요한 지점을 제안받을 수 있습니다. **Spec은 단일 객체의 문서일 뿐 아니라, 여러 모듈 사이의 규칙을 맞춰 보는 기준점**이 됩니다.

### DDD Building Block 단위라서 Spec이 쓰기 쉽다

"주문 서비스의 Spec을 써줘"라고 하면 범위가 너무 넓습니다. 무엇부터 써야 할지 막막합니다. 하지만 "Name Value Object의 Spec을 써줘"라고 하면? 범위가 좁고, 규칙이 명확하고, 한 페이지 안에 끝납니다. DDD가 도메인을 작은 Building Block으로 분해하기 때문에, 각 Block에 대한 Spec을 쓰는 것이 자연스럽습니다.

이 차이는 데이터 위주 설계와 비교하면 더 분명해집니다. 데이터 위주 설계에서는 비즈니스 로직이 Service 레이어에 넓게 퍼져 있어서, 하나의 규칙을 설명하려 해도 여러 조건과 흐름을 한꺼번에 끌고 와야 합니다. 반면 DDD의 Building Block은 작고 주체적인 단위로 나뉘어 있기 때문에, 각 객체의 책임을 따로 생각하고 모델링한 뒤 Spec으로 정리하기가 훨씬 쉽습니다.

**큰 서비스 하나의 Spec은 쓰기 어렵지만, 작은 Value Object 하나의 Spec은 대화 몇 번이면 완성됩니다.**

그렇다면 실제로 SDD를 어떻게 사용하는지 보겠습니다. 가장 작은 단위인 Value Object부터 시작해, AI와 대화하며 규칙을 확인하고 Spec 문서로 정리하는 흐름을 하나씩 따라가 보겠습니다.

## Name Spec 정의 — AI와의 대화

### 대화 시작

아래 대화는 실제 흐름을 이해하기 쉽도록 정리한 예시입니다. 실제 작업에서는 질문의 순서가 달라질 수 있고, 중간에 규칙을 다시 수정하는 일도 자주 있습니다.

> **나:** 고객의 이름을 표현하는 Name Value Object가 필요해. Spec을 같이 정의해 보자.

> **AI:** 좋습니다. Name에 대해 몇 가지 확인하겠습니다.
>
> 1. 빈 문자열이 허용되나요?
> 2. 최대 길이 제한이 있나요?
> 3. 앞뒤 공백은 어떻게 처리하나요?
> 4. 특수문자나 숫자가 포함될 수 있나요?

> **나:** 빈 문자열은 안 돼. 길이는 50자 이하. 앞뒤 공백은 자동으로 잘라줘. 특수문자는 일단 허용할게.

> **AI:** 정리하면 이렇습니다.
>
> - 빈 문자열 불가
> - 50자 초과 불가
> - 앞뒤 공백 자동 제거(trim)
> - 특수문자, 숫자 허용
>
> 한 가지 더 — 공백만으로 이루어진 문자열(예: `"   "`)은 trim 후 빈 문자열이 되는데, 이것도 빈 문자열 불가 규칙에 걸리나요?

> **나:** 맞아. trim 먼저 하고 빈 문자열 체크하면 되겠지.

> **AI:** 그러면 순서가 중요합니다. 검증 흐름을 정리하면:
>
> 1. 입력값 trim
> 2. trim된 결과가 빈 문자열인지 검사
> 3. trim된 결과가 50자 초과인지 검사
> 4. 통과하면 Name 생성
>
> 이 순서가 맞나요?

> **나:** 맞아. 그대로 가자.

> **나:** 좋아. 지금까지 정리한 내용을 Spec 문서로 만들어줘. `docs/specs/value-objects/name.md`에 저장해줘.

> **AI:** `docs/specs/value-objects/name.md`를 생성했습니다.

대화의 산출물이 프로젝트 안에 파일로 남습니다. Spec 문서는 프로젝트의 `docs/specs/` 아래에서 DDD 분류에 맞게 나누어 저장하는 것을 추천합니다. 코드와 함께 버전 관리되고, 나중에 AI에게 context로 전달하기도 쉽습니다.

```
project/
├── docs/
│   └── specs/
│       └── value-objects/
│           ├── name.md
│           ├── reward-points.md
│           └── email.md
├── src/
│   └── domain/
│       └── value-objects/
│           ├── name.ts
│           ├── reward-points.ts
│           └── email.ts
└── tests/
    └── domain/
        └── value-objects/
            ├── name.test.ts
            ├── reward-points.test.ts
            └── email.test.ts
```

이번 예제에서는 `docs/specs/value-objects/`에 Spec, `src/domain/value-objects/`에 구현, `tests/domain/value-objects/`에 테스트를 둡니다. Value Object부터 시작하는 단계에서는 이런 구조가 특히 직관적입니다. Spec을 수정하면 테스트를 수정하고, 테스트를 수정하면 구현을 수정하는 흐름이 디렉토리 구조에서도 드러납니다.

### 왜 객체별로 Spec을 나누는가

이 구조를 추천하는 이유는, DDD의 각 Building Block이 저마다 **주체적인 규칙과 책임**을 가지기 때문입니다. `Name`, `Email`, `RewardPoints`처럼 작은 Value Object도 각각 따로 확인해야 할 생성 규칙, 불변 조건, 동등성이 있습니다. 이후 `Entity`, `Aggregate`, `Domain Service`로 넘어가면 각 객체가 책임지는 요구사항과 기능은 더 분명하게 나뉘게 됩니다. 그래서 DDD로 설계한 객체들은 가능하면 **Building Block마다 하나의 Spec**을 두고 관리하는 편이 좋습니다. 하나의 큰 문서 안에 여러 객체의 규칙을 섞어 두면, 나중에 수정할 때 어떤 규칙이 어느 객체의 책임인지 흐려지기 쉽습니다. 반대로 객체별로 Spec을 분리해 두면, 해당 객체의 요구사항과 기능을 독립적으로 검토하고, 테스트와 구현도 같은 단위로 맞춰 가기가 쉬워집니다.

이렇게 해 두면 각 DDD Building Block마다 `Spec -> Test -> 구현`이 한 짝으로 대응하게 됩니다. 나중에 특정 객체를 다시 이해해야 하거나, 한동안 손대지 않았던 기능의 context를 복원해야 할 때도 이 세 가지를 함께 보면 됩니다. Spec은 "무엇을 지켜야 하는가"를 보여 주고, Test는 "무엇을 검증하는가"를 보여 주며, 구현은 "그 규칙을 실제로 어떻게 담았는가"를 보여 줍니다. 즉 객체 단위로 문서와 테스트와 코드가 함께 정렬되어 있을수록, 이해와 변경이 훨씬 쉬워집니다.

### 완성된 Name Spec

`docs/specs/value-objects/name.md`의 내용입니다.

```markdown
## Name (Value Object)

### 생성 규칙
- 앞뒤 공백은 무시한다
- 정규화한 결과가 비어 있으면 유효하지 않다
- 정규화한 결과가 50자를 초과하면 유효하지 않다

### 불변 조건
- Name의 value는 항상 비어 있지 않다
- Name의 value는 항상 50자 이하이다
- Name의 value에는 앞뒤 공백이 없다

### 동등성
- 두 Name은 value가 같으면 같다
```

눈여겨볼 것은, **코드가 한 줄도 없다는 점** 입니다. 순수하게 규칙만 정리했습니다. 이 Spec이 있으면 구현은 Ch.6에서 TDD로 옮기면 됩니다.

그리고 대화가 없었다면 "trim 먼저, 검증 나중"이라는 순서를 명시적으로 적었을까요? "공백만으로 이루어진 문자열"이라는 엣지 케이스를 떠올렸을까요? AI와의 대화가 Spec의 정밀도를 높여줍니다.

## RewardPoints Spec 정의 — AI와의 대화

Name보다 한 단계 더 복잡한 케이스입니다. RewardPoints에는 **행동(method)** 이 있기 때문입니다.

### 대화 시작

> **나:** 고객의 적립 포인트를 표현하는 RewardPoints Value Object가 필요해. Ch.2에서는 생성과 add 정도만 봤는데, 이번에는 조금 더 실제 설계에 가깝게 Spec을 정의하자.

> **AI:** 기본적인 것부터 확인하겠습니다.
>
> 1. 음수 포인트가 가능한가요?
> 2. 소수점 포인트가 가능한가요? (예: 1.5 포인트)
> 3. 최대값 제한이 있나요?

> **나:** 음수는 안 돼. 정수만 허용. 최대값은 일단 제한 없어.

> **AI:** 알겠습니다. 행동에 대해서도 정리해야 합니다.
>
> 1. 포인트 적립(add)이 가능한가요?
> 2. 포인트 차감(deduct/subtract)이 가능한가요?
> 3. 차감 시 잔액보다 많이 차감하면 어떻게 되나요?

> **나:** 적립은 당연히 가능. 차감도 가능한데, 잔액보다 많이 차감하면 에러를 던져야 해.

> **AI:** 차감 결과가 음수가 되면 에러라는 뜻이군요. 그러면 차감 시에도 "결과가 0 이상"이라는 불변 조건이 유지되어야 합니다.
>
> 한 가지 더 — 0 포인트 적립이나 0 포인트 차감은 허용하나요? 비즈니스적으로 의미가 있을까요?

> **나:** 음... 0 포인트 적립은 의미 없으니까 막는 게 맞겠다. 적립/차감 금액은 양수(1 이상)만 허용하자.

> **AI:** 좋습니다. 정리하면:
>
> - `add(points)`: points는 1 이상의 정수. 현재 값에 더함
> - `deduct(points)`: points는 1 이상의 정수. 현재 값에서 뺌. 결과가 음수면 Error
>
> 그리고 `add`와 `deduct`의 인자가 `number`가 아니라 `RewardPoints`여야 할까요? 아니면 `number`를 받을까요?

> **나:** 좋은 질문이야. Ch.2에서 Value Object끼리 연산하는 게 핵심이었으니까, `RewardPoints`를 받는 게 일관적이겠다.

> **나:** 좋아. `docs/specs/value-objects/reward-points.md`에 Spec 문서로 정리해줘.

> **AI:** `docs/specs/value-objects/reward-points.md`를 생성했습니다.

### 완성된 RewardPoints Spec

```markdown
## RewardPoints (Value Object)

### 생성 규칙
- 0 이상의 정수만 유효하다

### 행동
- 다른 RewardPoints와 합산할 수 있다
- 다른 RewardPoints를 차감할 수 있다
- 적립/차감에 사용되는 포인트는 1 이상이어야 한다
- 차감 결과가 음수가 되면 유효하지 않다

### 불변 조건
- RewardPoints의 value는 항상 0 이상의 정수이다

### 동등성
- 두 RewardPoints는 value가 같으면 같다
```

Name과 비교해 보세요. **행동(method) 섹션**이 추가되었습니다. Value Object가 단순한 값 검증을 넘어 비즈니스 행동까지 품고 있으니, Spec에도 그것이 반영됩니다. 그리고 "0 포인트 적립/차감을 막을 것인가"라는 질문 — 이것은 코드를 짜면서 생각하기 어려운 비즈니스 결정입니다. Spec을 먼저 정리했기 때문에, 구현 전에 이 결정을 내릴 수 있었습니다.

## Email Spec 정의 — AI와의 대화

Email에서 AI와의 대화가 특히 흥미로웠던 부분은 **정규화 정책의 범위를 결정하는 과정**이었습니다.

> **나:** 고객의 이메일을 표현하는 Email Value Object가 필요해. Spec을 정의하자.

> **AI:** 이메일 정규화 범위를 정해야 합니다.
>
> 1. 대소문자: `User@Example.COM` → `user@example.com`?
> 2. 앞뒤 공백 제거?
> 3. Gmail의 dot notation (`j.doe@gmail.com` = `jdoe@gmail.com`) 같은 것도 처리하나요?
> 4. 플러스 주소(`user+tag@example.com`)는 `user@example.com`과 같은 것으로 볼 건가요?
> 5. 국제화 도메인(IDN, 예: `user@예시.한국`)은 지원하나요?

> **나:** 대소문자는 소문자로 통일. 공백 제거는 당연히. Gmail dot notation, 플러스 주소, 국제화 도메인까지는 안 가도 돼. 지금 단계에서는 범위를 좁게 가져가자.

> **AI:** 알겠습니다. 정규화 범위를 정리하면:
>
> - 앞뒤 공백 제거
> - 소문자 통일
> - 그 외 provider별 특수 규칙은 처리하지 않음
>
> 이 범위가 맞나요?

> **나:** 맞아. `docs/specs/value-objects/email.md`에 정리해줘.

> **AI:** `docs/specs/value-objects/email.md`를 생성했습니다.

여기서 주목할 점은, AI가 제안한 여러 정규화 옵션 중 **의식적으로 범위를 제한하는 결정**을 내렸다는 것입니다. SDD에서 Spec을 쓴다는 것은 "무엇을 할 것인가"를 정하는 것이면서 동시에 "무엇을 하지 않을 것인가"를 명시하는 것이기도 합니다. 이런 결정은 팀마다 다릅니다. 중요한 건 **Spec에 명시적으로 기록된다는 것**입니다.

### 완성된 Email Spec

```markdown
## Email (Value Object)

### 생성 규칙
- 앞뒤 공백은 무시한다
- 대소문자는 구분하지 않고 소문자로 정규화한다
- 유효한 이메일 형식이어야 한다

### 불변 조건
- Email의 value는 항상 유효한 이메일 형식이다
- Email의 value는 항상 소문자이다
- Email의 value에는 앞뒤 공백이 없다

### 동등성
- 두 Email은 value가 같으면 같다
```

## Value Object Spec에서 자주 확인하는 항목

세 개의 Spec을 나란히 놓아 보니, 반복해서 확인하게 되는 항목이 보입니다.

| Value Object | 생성 규칙 | 행동 | 불변 조건 | 동등성 |
|---|---|---|---|---|
| **Name** | 앞뒤 공백 무시, 비어 있으면 불가, 50자 초과 불가 | — | 비어 있지 않음, 50자 이하, 앞뒤 공백 없음 | value 동일 |
| **RewardPoints** | 0 이상의 정수만 유효 | add, deduct | 0 이상 정수 | value 동일 |
| **Email** | 앞뒤 공백 무시, 소문자 정규화, 유효한 형식이어야 함 | — | 유효한 형식, 소문자, 앞뒤 공백 없음 | value 동일 |

Value Object를 다룰 때는 이 네 가지를 자주 확인하게 됩니다. 다만 이것이 고정된 템플릿은 아닙니다. 어떤 Value Object는 행동이 거의 없을 수도 있고, 어떤 Spec은 예외 상황이나 정규화 정책을 더 자세히 적는 편이 자연스러울 수 있습니다. 중요한 것은 형식을 맞추는 것이 아니라, **객체의 규칙을 빠뜨리지 않고 명확하게 적는 것**입니다.

DDD에서 Value Object는 "의미 있는 값을 표현하는, 불변이고, 동등성으로 비교하는 객체"입니다. 그래서 Spec을 쓰다 보면 자연스럽게 생성 규칙, 불변 조건, 동등성 같은 질문이 반복해서 등장합니다. **Spec을 쓰다 보면 자연스럽게 DDD Building Block의 형태가 나타납니다.** 이것이 DDD와 SDD가 잘 맞는 이유입니다.

### Spec은 Test의 기준이 된다

여기서 한 걸음 더 나가면 바로 TDD로 이어집니다. Spec이 있으면 무엇을 검증해야 하는지가 이미 정리되어 있기 때문에, 이를 준수하는 테스트를 만들기가 훨씬 쉬워집니다. 게다가 DDD의 도메인 모델은 프레임워크에 덜 의존하는 순수한 객체로 유지하기 좋기 때문에, 테스트 역시 더 빠르고 단순하게 작성할 수 있습니다. 하지만 더 중요한 것은, 이 테스트가 이후 변경 작업의 기준점이 된다는 점입니다. AI와 함께 개발하다 보면 구현을 한 번 만들고 끝나지 않습니다. 요구사항이 바뀌고, 규칙이 추가되고, 기존 코드를 계속 수정하게 됩니다. 이때 테스트는 "무엇이 깨지면 안 되는가"를 보여주는 가이드라인 역할을 합니다. AI가 코드를 수정하더라도 테스트가 통과하지 않으면, 적어도 기존 규칙을 어겼다는 사실을 바로 알 수 있습니다.

즉 Spec은 변경의 기준을 만들고, Test는 그 기준이 실제로 지켜지는지를 계속 감시합니다. AI와 반복적으로 협업할수록 이 역할은 더 중요해집니다.

### Spec과 Code는 항상 함께 갱신되어야 한다

여기서 한 가지 운영 원칙이 더 중요합니다. Spec을 문서로 남겼다면, 그 Spec과 실제 코드가 항상 서로 맞도록 관리해야 합니다. 좋은 흐름은 대개 이렇습니다.

1. Spec을 논의하고 확정한다
2. 그 Spec을 바탕으로 테스트 케이스를 논의하고 확정한다
3. 테스트를 구현한다
4. 테스트를 통과하도록 구현한다

이 순서를 따르지 않고 코드만 먼저 바꾸면, 문서로 남은 Spec이 빠르게 낡아 버릴 수 있습니다. 문제는 여기서 끝나지 않습니다. 나중에 누군가 Spec을 읽고 새 기능을 구현하거나, AI에게 Spec을 context로 넘겨 다시 작업하게 될 때, Spec과 Code가 어긋나 있으면 잘못된 방향으로 구현이 이어질 수 있습니다.

즉 Spec은 한 번 쓰고 잊는 문서가 아니라, 코드와 함께 계속 맞춰 가야 하는 기준입니다. AI와 함께 여러 번 수정해 나갈수록, 이 sync를 유지하는 일이 더 중요해집니다.

## 참고: SDD를 도와주는 도구들

이번 글에서는 AI와 직접 대화하며 Spec을 정의하는 과정을 보여드렸습니다. 이 방식만으로도 충분히 효과적이지만, SDD 워크플로우를 더 체계적으로 지원하는 도구들도 있습니다. 다만 중요한 것은 도구 자체보다, **구현 전에 Spec을 먼저 정의하는 습관**입니다.

- **Superpower**: AI와의 대화 흐름을 `Plan -> Spec -> Implementation` 순서로 정리해 주는 도구입니다. Spec 문서를 만들고, 그 Spec을 바탕으로 다음 구현 단계까지 이어 가는 과정을 구조화할 때 유용합니다.
- **OpenSpec**: Spec 문서의 형식 자체를 더 일관되게 관리하고 싶을 때 참고할 수 있는 오픈소스 규약입니다. 생성 규칙, 행동, 불변 조건처럼 반복되는 구조를 표준화해 팀 내 문서 스타일을 맞추는 데 도움이 됩니다.
- **GitHub Spec Kit**: GitHub 환경에서 Spec 기반 워크플로우를 지원하는 도구입니다. Spec 문서와 구현 작업을 연결해 관리하고, 변경된 Spec이 어떤 코드와 테스트에 영향을 주는지 추적하는 데 유용합니다.
