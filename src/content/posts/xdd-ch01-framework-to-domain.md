이 글은 **[SDD+TDD+DDD] 시리즈**의 첫 번째 글입니다. <br />

혹시 새로운 프레임워크가 나올 때마다 쫓아가느라 지치고 있지는 않으신가요? 프레임워크는 계속 바뀌는데, 정작 내가 만드는 소프트웨어의 본질은 무엇인지 흐릿하게 느껴진 적은 없으신가요?
MVC를 따라 데이터를 중심으로 개발하다 보니 비즈니스 로직이 여러 곳에 흩어지고, 무엇 하나 고치려 해도 여러 파일을 뒤져야 했던 경험은 없으신가요?

저 역시 같은 고민을 오래 해왔습니다. Spring, Next.js, React, NestJS 같은 프레임워크 중심 개발을 해 오다가, AI 시대를 맞아 점점 더 분명해진 것이 있었습니다. **개발자의 본질적 역할은 도메인에 있고, 프레임워크는 AI의 도움을 크게 받을 수 있는 영역이 되어가고 있다**는 점입니다.

그래서 이 시리즈에서는 다시 TDD와 DDD를 이야기하려 합니다. 여기에 AI와 협업하는 흐름을 담기 위해 SDD까지 함께 묶었습니다.

이 글에서 말하는 SDD+TDD+DDD는 다음 세 가지를 함께 부르는 표현입니다.

* SDD: Spec-Driven Development
* TDD: Test-Driven Development
* DDD: Domain-Driven Design

이 시리즈는 세 파트로 구성됩니다.

- **Part 1** (Ch.1~8): SDD+TDD+DDD 입문 — 왜 도메인 중심인가, DDD 기본 어휘, top-down 설계, SDD와 TDD 첫 체험.
- **Part 2** (Ch.9~15): DDD Building Blocks 상세 — 본편 도메인을 기준으로 각 Building Block을 개념 + SDD + TDD로 완결.
- **Part 3** (Ch.16~): DDD 응용 및 고급 주제 — Hexagonal Architecture, 프론트엔드 DDD, Event Sourcing/CQRS, FP와 DDD 등.

## 프레임워크 중심 + 데이터 중심 개발, 무엇이 문제인가

### 끝없는 프레임워크 학습의 반복

돌이켜 보면 저의 개발 공부 역사는 곧 프레임워크 학습의 역사였습니다. 백엔드를 시작할 때는 Spring을 배웠고, Node.js 생태계로 넘어오면서 Express를 익혔고, 좀 더 구조화된 개발을 위해 NestJS를 공부했습니다. 프론트엔드도 비슷했습니다. jQuery로 시작해 React를 배웠고, 이제는 Next.js가 사실상 표준처럼 자리 잡았습니다. 하나를 어느 정도 익히면 또 새로운 것이 나오고, 그걸 다시 처음부터 배워야 하는 순환이 이어졌습니다.

특히 고통스러운 건 메이저 버전 업그레이드입니다. 같은 프레임워크라도 메이저 버전이 올라가면 API가 크게 달라지고, 기존 코드를 대규모로 수정해야 할 때가 있습니다. Angular 1에서 Angular 2로의 전환을 경험한 분이라면, 혹은 Next.js의 Pages Router에서 App Router로의 마이그레이션을 겪은 분이라면 이 부담을 잘 아실 겁니다. 어제까지 익숙했던 API가 오늘 deprecated 되어 있는 것을 발견했을 때의 허탈함도 낯설지 않습니다. 프레임워크 지식의 유통기한은 생각보다 짧습니다.

문제는 이 과정에서 "프레임워크를 배우는 것이 곧 개발을 배우는 것"이라는 착각에 빠지기 쉽다는 점입니다. 새로운 프레임워크의 컨벤션과 API를 익히면 실력이 늘었다고 느끼기 쉽습니다. 하지만 **그 프레임워크 없이 비즈니스 문제를 모델링하고 설계하는 능력은 별도로 훈련되지 않습니다.** 프레임워크는 도구인데, 우리는 어느새 도구의 구조에 사고를 맞추고 있었습니다.

### 데이터 중심 프로그래밍의 한계

그런데 프레임워크 중심 학습의 문제는 단순히 "새로운 걸 계속 배워야 해서 피곤하다"는 데서 끝나지 않습니다. 더 근본적인 문제가 있습니다. 프레임워크가 제시하는 패턴을 따라가다 보면, 자연스럽게 데이터 중심 프로그래밍에 익숙해진다는 점입니다.
 
대부분의 백엔드 프레임워크는 MVC 혹은 그 변형 패턴을 기본으로 제공합니다. Controller가 요청을 받고, Service가 비즈니스 로직을 처리하고, Repository가 데이터를 저장하는 구조입니다. 이 패턴 자체가 문제는 아닙니다. 다만 이 구조를 그대로 따라가다 보면 주체적인 도메인 객체는 사라지고, 그저 데이터를 담는 그릇, 즉 Anemic Domain Model이 되기 쉽습니다. DB Entity나 DTO는 필드만 가지고 있고, 실제 비즈니스 규칙은 Service 레이어에 절차적으로 나열됩니다.

처음에는 괜찮습니다. 코드가 적을 때는 Service 하나에 로직을 모아 두어도 충분히 관리할 수 있습니다. 하지만 서비스가 성장하면 이야기가 달라집니다. "주문 생성 시 재고를 확인하고, 할인 정책을 적용하고, 결제를 처리하는" 로직이 OrderService에도 있고, DiscountService에도 있고, PaymentController에도 일부 흩어져 있는 상황이 벌어집니다. 그러면 "이 로직은 어디서 수정하지?"라는 질문에 자신 있게 답하기 어려워집니다. 비즈니스 로직의 응집도가 낮고, 여러 곳에 분산되어 있기 때문입니다.

프론트엔드도 다르지 않습니다. React나 Next.js에서 컴포넌트를 만들다 보면 `useEffect` 안에서 상태 전이를 판단하고, 조건 분기로 비즈니스 규칙을 처리하는 코드가 자연스럽게 생깁니다. "이 주문은 취소 가능한 상태인가?"를 컴포넌트 안에서 직접 판단하고, 비슷한 로직이 다른 페이지의 커스텀 훅에도 중복되어 들어갑니다. 백엔드에서 Service에 로직이 흩어지듯, 프론트엔드에서는 컴포넌트와 훅에 로직이 흩어집니다. 도메인 객체 없이 API 응답 데이터를 직접 조작하는 패턴도 흔합니다. 백엔드든 프론트엔드든 근본 원인은 비슷합니다.

결국 인과 관계는 분명합니다. **프레임워크 중심 학습**은 프레임워크가 제공하는 컨벤션과 계층 구조에 사고를 맞추게 만들고, 이것은 자연스럽게 **데이터 중심 프로그래밍**으로 이어지기 쉽습니다. 도메인 객체는 데이터만 들고 있는 껍데기가 되고, 비즈니스 로직은 Service 곳곳에 **흩어집니다**. 코드가 늘어날수록 "어디를 고쳐야 하는지"를 파악하는 것 자체가 일이 됩니다. 프레임워크를 아무리 잘 알아도 도메인을 제대로 모델링하지 않으면, 유지보수 비용은 빠르게 커집니다.

오해하지 말아야 할 것은, 프레임워크가 중요하지 않다는 이야기가 아니라는 점입니다. 프레임워크는 당연히 필요하고, 잘 활용하면 생산성을 크게 높여줍니다. 문제는 비중의 불균형입니다. 우리의 시간과 에너지 대부분을 프레임워크 학습에 쏟고, 정작 "이 소프트웨어가 풀어야 할 비즈니스 문제"를 모델링하는 데는 거의 투자하지 않는 현실 말입니다. 프레임워크를 버리자는 것이 아니라, 도메인에 더 집중하자는 것입니다.

이 문제를 오랫동안 체감하면서도 명확한 대안을 찾지 못하고 있었습니다. 그런데 전환점이 된 계기가 있었습니다.

## AI 시대, 역할이 뒤바뀌다

AI 코드 생성 도구들이 등장했습니다. Copilot, Cursor, Claude Code 같은 도구들이 개발 현장에 빠르게 스며들었고, 저도 자연스럽게 사용하게 되었습니다. 이 도구들을 쓰면 쓸수록 한 가지 사실이 분명해졌습니다. AI는 프레임워크 작업의 많은 부분에서 매우 강력합니다.

NestJS로 CRUD endpoint를 만드는 일, React 컴포넌트의 보일러플레이트를 작성하는 일, DB migration 스크립트를 생성하는 일. 이런 작업에서 AI는 빠르게 초안을 만들고 관용적인 패턴을 적용하는 데 강합니다. API 문서를 뒤적이며 파라미터 순서를 확인하고, 오타를 찾고, 컨벤션에 맞는지 검토하는 시간이 크게 줄어듭니다. 이 경험이 반복되면서 한 가지 생각이 선명해졌습니다. 내가 프레임워크 API를 외워서 얻는 가치는 예전보다 빠르게 낮아지고 있다는 것. 이전에는 특정 프레임워크에 대한 깊은 지식이 곧 개발자의 경쟁력이라고 여겼는데, 그 전제가 흔들리고 있었습니다.

그런데 흥미로운 지점은 그 반대편에 있었습니다. AI가 놀라울 정도로 잘하는 영역이 있는 반면, 맥락 없이는 쉽게 놓치는 영역도 있었습니다. 바로 **우리 비즈니스의 고유한 도메인 문제**입니다.

* "스프린트 종료 12시간 이내에 생성된 태스크는 완료율 집계에서는 제외되지만, 번다운 차트에는 포함된다"는 규칙
* "외부 협력사에게 할당된 태스크는 우선순위가 P1이어도 당일 재할당할 수 없고, 담당 부서장의 승인 이력이 있어야만 이동할 수 있다"는 제약
* "보안 등급이 '고객 데이터 포함'인 태스크는 작성 팀과 승인 팀이 같더라도, 배포 단계로 올리기 전에 운영팀 체크리스트를 반드시 통과해야 한다"는 규칙

도메인의 고유한 문제일수록 일반적인 AI가 알기 어렵습니다. 이것은 일반 지식이 아니라, 우리 팀이 오랜 시간 비즈니스를 운영하며 쌓아 온 맥락이고, 수많은 논의와 실패를 통해 정제된 규칙이기 때문입니다. 아무리 뛰어난 AI라도 사전 정보와 명시적인 맥락 없이는 우리만의 문제를 대신 정의해 줄 수는 없습니다. 도메인 지식은 여전히 인간이 책임져야 할 영역으로 남아 있었습니다.

이 두 가지 관찰이 겹치면서 하나의 결론에 도달했습니다. 개발자의 역할이 재정의되고 있다는 것입니다. 프레임워크 레이어의 작업, 즉 보일러플레이트 생성, API 연결, 데이터 접근 계층 구성 같은 일은 AI에게 많이 위임할 수 있습니다. 반면 도메인 레이어의 작업, 즉 비즈니스 규칙을 이해하고 올바르게 모델링하며 그것이 정확히 동작하는지 검증하는 일은 개발자가 집중해야 할 본질적인 영역입니다.

내가 해결해야 하는 고유한 문제에 더 많은 시간을 써야 한다는 것. 이것이 AI 시대에 개발자가 가져야 할 방향이라는 확신이 생겼습니다. 이 인식이 DDD(Domain-Driven Design)라는 오래된 방법론과 만나면서, 구체적인 실천 방향도 보이기 시작했습니다.

## SDD → TDD → DDD: AI 시대의 개발 방법론

앞서 두 가지 문제를 확인했습니다.

* 프레임워크 중심 개발이 데이터 중심 설계로 이어진다는 것
* AI 시대에 개발자의 핵심 가치는 도메인에 있다는 것

이 두 인식이 만나는 지점에서 구체적인 방법론이 형성됩니다.

* SDD: Spec-Driven Development
* TDD: Test Driven Development
* DDD: Domain-Driven Design

이것이 이 시리즈에서 제안하는 AI 시대의 개발 방법론입니다.

### 도메인 중심 모델링 — 비즈니스 로직을 도메인에 응집

데이터 중심 프로그래밍에서는 도메인 객체가 데이터 필드만 가진 빈 그릇이었습니다. 비즈니스 로직은 Service 레이어에 절차적으로 흩어져 있었고, 규칙이 어디에 있는지 파악하는 것 자체가 일이었습니다. DDD는 이 구조를 뒤집습니다. 도메인 객체가 데이터뿐 아니라 행위(behavior)도 함께 가지도록 만드는 것입니다. "이 콘텐츠는 지금 완료 처리할 수 있는가?"라는 판단은 Service가 아니라 Content 객체 자신이 합니다. 비즈니스 로직이 도메인 객체 안에 응집되면 코드를 수정할 위치가 자명해집니다. 응집도는 올라가고, 복잡도는 내려갑니다.

여기서 한 가지 덧붙일 점이 있습니다. DDD를 말한다고 해서 "모든 것을 클래스로 만들자"는 뜻은 아닙니다. OOP와 Functional Programming은 대립적인 관계라기보다, 도메인을 표현하는 서로 다른 도구에 가깝습니다. Entity나 Aggregate처럼 정체성과 생명주기를 가지며 상태 변경의 규칙을 지켜야 하는 대상은 객체로 모델링하는 편이 자연스럽습니다. 반면 계산, 변환, 정책 판단처럼 부수 효과 없이 입력을 받아 결과를 만드는 로직은 순수 함수로 표현하는 편이 더 명확할 때가 많습니다. 중요한 것은 OOP냐 FP냐의 진영이 아니라, 도메인의 성격에 맞는 표현을 선택해 규칙과 책임의 경계를 분명히 하는 것입니다.

DDD에서는 이 도메인 모델을 구성하는 핵심 Building Block들이 있습니다.

- **Value Object**: 의미 있는 값을 표현하며, 불변이고, 동등성(equality)으로 비교합니다. 예를 들어 `ContentUrl`, `Tag`, `ConsumptionStatus` 같은 것들입니다.
- **Entity**: 고유한 ID를 가지며 생명주기를 가진 객체입니다. `Content` 같은 대상이 여기에 해당합니다.
- **Aggregate**: 일관성(consistency)의 경계입니다. Aggregate 외부에서 내부 객체를 직접 수정할 수 없으며, 반드시 Aggregate Root를 통해야 합니다. `Content`가 대표적인 예입니다.
- **도메인 서비스**: 하나의 Entity에 귀속시키기 어려운, 여러 Entity에 걸친 비즈니스 로직을 담당합니다.
- **Repository**: 도메인 객체의 저장과 조회를 위한 인터페이스입니다. 인터페이스는 도메인 레이어에 정의하되, 실제 구현은 인프라 레이어에 둡니다.

이 Building Block들은 시리즈를 진행하면서 하나씩 Spec 정의 → TDD 테스트 작성 → 구현 사이클로 만들어 갑니다.

### 프레임워크와 분리된 도메인 모델 — TDD가 쉬워지다

DDD의 도메인 모델은 프레임워크에 의존하지 않는 순수한 객체로 구현할 수 있습니다. NestJS의 데코레이터도, Spring의 어노테이션도, React의 훅도 필요 없습니다. 도메인 코드는 특정 프레임워크에 끌려가지 않도록 유지할 수 있습니다.

이것이 왜 중요할까요? 외부 의존성이 없는 순수 객체는 DB나 HTTP 없이 도메인 로직만 바로 테스트할 수 있게 해 줍니다. DB 연결을 흉내 내거나 HTTP 요청을 가로채는 상황이 줄어들고, 객체를 생성한 뒤 메서드를 호출하고 결과를 검증하는 방식으로 빠르게 피드백을 얻을 수 있습니다. 즉, 도메인 레이어에서는 TDD를 적용하기가 훨씬 쉬워집니다.

그리고 프레임워크가 바뀌어도 도메인 코드는 상대적으로 안정적으로 유지됩니다. Express에서 NestJS로 전환하든, REST에서 GraphQL로 바꾸든 도메인 레이어는 직접적인 영향을 덜 받습니다. 프레임워크는 도메인을 감싸는 어댑터에 가깝고, 도메인이 중심이 되는 구조입니다. 이것이 프레임워크 비의존 아키텍처의 핵심입니다.

### SDD (Spec-Driven Development) : AI와 함께 Spec 먼저

여기에 AI 시대의 새로운 요소를 하나 더합니다. SDD, Spec-Driven Development입니다.

SDD는 구현에 앞서 AI와의 대화를 통해 Spec을 먼저 정리하는 개발 방식입니다. BDD(Behavior-Driven Development)처럼 행동 명세를 중심으로 개발하지만, 여기서는 AI가 Spec 정리를 보조한다는 차이가 있습니다. 개발자는 AI에게 도메인 규칙을 설명하고, AI는 이해한 바를 Spec 형태로 정리합니다. 개발자는 그 결과를 검증하고 보완합니다. 이 과정에서 암묵적이던 도메인 지식이 명시적인 명세로 바뀝니다.

Spec이 정의되면 TDD 테스트 케이스도 훨씬 자연스럽게 도출됩니다. Spec의 각 항목이 테스트 케이스의 후보가 되기 때문입니다. 전체 워크플로우는 다음과 같습니다.

**(AI와) Spec 정의 -> TDD 테스트 작성 -> 도메인 구현 -> 프레임워크 연동 (AI 보조)**

이 워크플로우의 장점은 인간의 영역과 AI의 영역이 비교적 자연스럽게 나뉜다는 점입니다.

* 도메인 지식을 바탕으로 Spec을 확정
* 그 Spec이 정확히 구현되었는지 검증할 테스트 케이스를 정의

이 부분은 개발자의 몫입니다. 반면

* 테스트 케이스가 확정된 이후 테스트 코드의 초안을 작성하는 것
* 테스트를 통과시키기 위해 Spec에 따라 구현을 구체화하는 것
* 도메인이 완성된 이후 프레임워크 연동, 보일러플레이트 생성, 인프라 코드 작성

AI가 효율적으로 보조할 수 있는 영역입니다. 각자가 잘하는 일에 집중하는 구조입니다.

### 닫힌 루프가 주는 통제력 : Spec → Test → Implementation

여기서 한 가지 더 주목할 점이 있습니다. Spec을 명시적으로 정의하고, 이를 검증하는 테스트 케이스를 먼저 확정한 뒤, 그 테스트를 통과하도록 구현하는 방식은 AI가 작업할 범위를 분명하게 제한해 줍니다. 최종 목표가 Spec으로 규정되어 있고 통과해야 할 테스트가 명확하기 때문에 AI는 훨씬 좁고 구체적인 범위 안에서 코드를 작성하게 됩니다. 물론 이것이 사람의 검토를 완전히 대체하지는 않습니다. 다만 검토의 초점을 "모든 코드를 처음부터 읽는 일"에서 "Spec과 테스트가 잘 반영되었는지 확인하는 일"로 옮겨 줍니다.

### DDD에 SDD가 특히 잘 맞는 이유

그런데 왜 하필 DDD일까요? MVC에서는 비즈니스 로직을 모듈별 Service 레이어에 구현합니다. UserService, OrderService, PaymentService처럼 모듈 수만큼 큰 덩어리의 서비스가 만들어집니다. 이 큰 덩어리에 대해 Spec을 쓰고 테스트를 만들려면 범위가 넓고 의존성이 많아, Spec 정의도 테스트 작성도 까다롭습니다.

반면 DDD는 도메인을 훨씬 작고 구체적인 Building Block으로 분해합니다. 특정 값을 표현하는 Value Object, 고유한 ID 식별자를 가진 Entity, 여러 Entity를 묶어 일관성을 보장하는 Aggregate Root, 도메인 이벤트, 도메인 서비스, 애플리케이션 서비스. 각각이 명확한 책임과 경계를 가진 작은 단위입니다.

이 작은 단위가 SDD + TDD와 만나면 시너지가 커집니다. Building Block 하나하나에 대해 Spec을 정의하고, 그 Spec을 만족시키는 테스트 케이스를 작성하고, 테스트를 통과하도록 구현하는 사이클이 자연스럽게 돌아갑니다. "`ConsumptionStatus` Value Object의 상태 전이 규칙"처럼 범위가 좁고 명확한 Spec은 AI가 이해하기도 쉽고, 테스트 케이스도 자연스럽게 도출됩니다. 큰 Service 하나를 통째로 다루는 것과는 정밀함의 수준이 다릅니다.

### SDD가 돕는 AI 시대의 문제 — Context 복원

AI와 함께 개발할 때, 한 번에 끝나는 짧은 작업은 큰 문제가 없습니다. 대화 하나로 구현을 완료할 수 있기 때문입니다. 하지만 현실의 프로젝트는 며칠, 몇 달에 걸칩니다. 여기서 어려움이 드러납니다. 어제 작업한 맥락을 오늘 다시 복원하는 비용이 생각보다 크다는 점입니다. 시간이 지나면 AI는 이전 대화의 세부사항을 자연스럽게 잃고, 개발자는 그 맥락을 다시 설명해야 합니다. 일주일 전에 작업한 코드를 개선해야 하는 상황이라면, 그때의 맥락을 어떻게 복원해서 AI에게 전달할 수 있을까요?

SDD는 이 문제를 줄여 줍니다. 각 Building Block의 요구 사항과 Spec이 문서로 정리되어 있고, 이를 검증하는 테스트와 구현 코드가 함께 존재하기 때문입니다. 일주일 전 작업을 보강해야 한다면, 해당 Spec 문서를 AI와 공유하면 됩니다. "이 Spec으로 구현된 코드가 있는데, 여기에 이런 요구 사항을 추가해야 해"라는 식으로 맥락을 다시 불러올 수 있습니다. Spec 문서는 맥락 복원 비용을 낮춰 주고, 테스트는 그 변경이 맞는지 검증하는 장치가 됩니다. 즉, AI의 기억력보다 문서화된 Spec에 더 많이 의존하게 되는 것입니다.

DDD 설명부터 설계, SDD, TDD로 이어지는 실전 흐름은 Ch.3~6에서 다룹니다.

### 한 가지 주의할 점

DDD가 모든 상황에 적합한 것은 아닙니다. 비즈니스 로직이 단순하고 CRUD 위주의 작업이라면 도메인 모델을 정교하게 설계하는 것이 오히려 오버엔지니어링이 될 수 있습니다. DDD는 복잡한 비즈니스 규칙이 존재하고, 그 규칙이 시간에 따라 변화하고 성장하는 도메인에서 특히 힘을 발휘합니다. 프론트엔드에는 적용 방식에 제약이 있는 것도 사실이고, 어떤 도메인에서는 DDD가 최선의 선택이 아닐 수도 있습니다. 그런 경우에도 Spec을 먼저 정리하는 SDD와 순수 객체 중심으로 테스트 대상을 분리하는 접근은 충분히 가져갈 수 있습니다.

### SDD+TDD+DDD : 이 시리즈가 말하고 싶은 것

정리하겠습니다.
**SDD+TDD+DDD. 이것이 AI 시대의 개발 방법론입니다.**
 
SDD로 AI와 함께 Spec을 먼저 정의하고, TDD로 그 Spec을 테스트 케이스로 바꿔 검증하고, DDD로 비즈니스 로직을 도메인 객체에 응집시킵니다. 프레임워크는 도메인 바깥의 어댑터로 밀어내고, 그 영역은 AI에게 위임합니다. 개발자는 도메인에 집중합니다.

이 시리즈에서는 이 워크플로우를 콘텐츠 북마크와 소비 추적을 다루는 개인용 도구 도메인을 통해 처음부터 끝까지 구현해 봅니다.

다음 글에서는 같은 요구사항을 데이터 위주 vs 객체 위주 두 가지 사고방식으로 나란히 비교합니다.
