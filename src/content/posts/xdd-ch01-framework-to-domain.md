이 글은 **[SDD+TDD+DDD] 시리즈**의 첫 번째 글입니다. <br />

SDD+TDD+DDD는 다음을 하나로 묶은 표현입니다. 

* SDD: Spec-Driven Development
* TDD: Test-Driven Development
* DDD: Domain-Driven Design 

요즘 같은 AI와 함께 개발하는 시대에 구닥다리 같은 TDD와 DDD를 또 이야기하냐구요 ?

Spring, Next.js, React.js, Nest.js 와 같은 프레임워크 중심으로 개발해 온 한 개발자가, AI 시대를 맞아 도메인 중심 개발로 전환하게 된 이야기를 담고 있습니다. 결론부터 말하면, **AI 시대에 개발자의 본질적 역할은 Domain이고, 프레임워크는 AI의 도움을 많이 받을 수 있는 영역**이 되어가고 있다는 것입니다.

혹시 새로운 프레임워크가 나올 때마다 쫓아가느라 지치고 있지는 않으신가요? 프레임워크는 계속 바뀌는데 정작 내가 만드는 소프트웨어의 본질은 뭔지 잘 모르겠다는 느낌이 든 적 있으신가요? 
MVC 이용한 데이터 위주로 프로그래밍으로 비즈니스 로직이 여러곳에 분산되어 있어서 뭐 하나 고치려면 여러개의 파일을 고치느라고 힘드신가요 ?

그렇다면 이 글이 도움이 될 수 있을 것 같습니다. 저 역시 같은 고민을 오래 해왔고, 그 고민 끝에 찾은 방향을 이 시리즈에서 공유하려 합니다.

## 프레임워크 중심 + 데이터 중심 개발, 어디가 문제인가

### 끝없는 프레임워크 학습의 반복

돌이켜 보면 저의 개발 공부 역사는 곧 프레임워크 학습의 역사였습니다. Backend를 시작할 때는 Spring을 배웠고, Node.js 생태계로 넘어오면서 Express를 익혔고, 좀 더 구조화된 개발을 위해 NestJS를 공부했습니다. Frontend도 마찬가지였습니다. jQuery로 시작해서 React를 배웠고, 이제는 Next.js가 사실상 표준처럼 자리 잡았습니다. 하나를 어느 정도 익히면 또 새로운 것이 나오고, 그걸 다시 처음부터 배워야 하는 순환이 끊임없이 반복되었습니다.

특히 고통스러운 건 Major version upgrade입니다. 같은 프레임워크인데 메이저 버전이 올라가면 API가 완전히 달라지고, 기존 코드를 대규모로 수정해야 합니다. Angular 1에서 Angular 2로의 전환을 경험한 분이라면, 혹은 Next.js의 Pages Router에서 App Router로의 마이그레이션을 겪은 분이라면 이 고통을 잘 아실 겁니다. 어제까지 열심히 외운 API가 오늘 deprecated 되어 있는 것을 발견했을 때의 그 허탈함. 프레임워크 지식의 유통기한은 생각보다 훨씬 짧습니다.

문제는 이 과정에서 "프레임워크를 배우는 것이 곧 개발을 배우는 것"이라는 착각에 빠지기 쉽다는 점입니다. 새로운 프레임워크의 Convention과 API를 익히면 실력이 늘었다고 생각하지만, **정작 그 프레임워크 없이 비즈니스 문제를 모델링하고 설계하는 능력은 크게 성장하지 않습니다. 프레임워크는 도구일 뿐인데, 우리는 어느새 도구에 종속된 개발자가 되어 있었습니다.**

### 데이터 중심 프로그래밍의 한계

그런데 프레임워크 중심 학습은 단순히 "새로운 걸 계속 배워야 해서 피곤하다"는 문제로 끝나지 않습니다. 더 근본적인 문제가 있습니다. 프레임워크가 제시하는 패턴을 따라가다 보면, 자연스럽게 데이터 중심 프로그래밍에 빠지게 된다는 것입니다.
 
대부분의 백엔드 프레임워크는 MVC 혹은 그 변형 패턴을 기본으로 제공합니다. Controller가 요청을 받고, Service가 비즈니스 로직을 처리하고, Repository가 데이터를 저장하는 구조. 이 패턴 자체는 나쁘지 않지만, 문제는 이 구조를 따르다 보면 주체적인 도메인 객체는 없고, 그저 데이터를 담는 그릇, 즉 Anemic Domain Model이 되기 쉽다는 점입니다. Entity나 DTO는 필드만 가지고 있고, 실제 비즈니스 규칙은 전부 Service 레이어에 절차적으로 나열됩니다.

처음에는 괜찮습니다. 코드가 적을 때는 Service 하나에 로직을 모아 두어도 충분히 관리 가능합니다. 하지만 서비스가 성장하면 이야기가 달라집니다. "주문 생성 시 재고를 확인하고, 할인 정책을 적용하고, 결제를 처리하는" 로직이 OrderService에도 있고, DiscountService에도 있고, PaymentController에도 일부 흩어져 있는 상황이 벌어집니다. "이 로직 어디서 수정하지?" 라는 질문에 자신 있게 답할 수 없게 됩니다. 비즈니스 로직의 응집도가 낮고, 여러 곳에 분산되어 있기 때문입니다.

프론트엔드도 다르지 않습니다. React나 Next.js에서 컴포넌트를 만들다 보면, useEffect 안에서 상태 전이를 판단하고, 조건 분기로 비즈니스 규칙을 처리하는 코드가 자연스럽게 생깁니다. "이 주문은 취소 가능한 상태인가?"를 컴포넌트 안에서 직접 판단하고, 비슷한 로직이 다른 페이지의 커스텀 훅에도 중복되어 들어갑니다. 백엔드에서 Service에 로직이 흩어지듯, 프론트엔드에서는 컴포넌트와 훅에 로직이 흩어집니다. 도메인 객체 없이 API 응답 데이터를 직접 조작하는 패턴 — 백엔드든 프론트엔드든, 근본 원인은 같습니다.

결국 인과 관계는 명확합니다. **프레임워크 중심 학습**은 프레임워크가 제공하는 Convention과 계층 구조에 사고를 맞추게 만들고, 이것은 자연스럽게 **데이터 중심 프로그래밍**으로 이어집니다. 도메인 객체는 데이터만 들고 있는 껍데기가 되고, 비즈니스 로직은 Service와 Controller 곳곳에 **흩어집니다**. 코드가 늘어날수록 "어디를 고쳐야 하는지"를 파악하는 것 자체가 일이 되고, 결국 **유지보수의 붕괴**로 이어집니다. 프레임워크를 아무리 잘 알아도, 도메인을 제대로 모델링하지 않으면 코드는 점점 통제 불능 상태에 빠지는 것입니다.

오해하지 말아야 할 것은, 프레임워크가 중요하지 않다는 이야기가 아니라는 점입니다. 프레임워크는 당연히 필요하고, 잘 활용하면 생산성을 크게 높여줍니다. 문제는 비중의 불균형입니다. 우리의 시간과 에너지의 대부분을 프레임워크 학습에 쏟고, 정작 "이 소프트웨어가 풀어야 할 비즈니스 문제"를 모델링하는 데는 거의 투자하지 않는 현실. 프레임워크를 버리자는 것이 아니라, 도메인에 더 집중하자는 것입니다.

이 문제를 오랫동안 체감하면서도 명확한 대안을 찾지 못하고 있었습니다. 그런데 전환점이 된 계기가 있었습니다.

## AI 시대, 역할이 뒤바뀌다

AI 코드 생성 도구들이 등장했습니다. Copilot -> Cursor -> Claude Code 같은 도구들이 개발 현장에 빠르게 스며들기 시작했고, 저도 자연스럽게 사용하게 되었습니다. 그런데 이 도구들을 쓰면 쓸수록, 한 가지 불편한 사실을 직면하게 되었습니다. AI가 프레임워크 작업을 저보다 잘한다는 것입니다.

NestJS로 CRUD endpoint를 만드는 일, React 컴포넌트의 boilerplate를 작성하는 일, DB migration 스크립트를 생성하는 일. 이런 작업들을 AI에게 시켜 보면, 솔직히 제가 직접 하는 것보다 빠르고 정확합니다. API 문서를 뒤적이며 파라미터 순서를 확인하고, 오타를 찾아 헤매고, Convention에 맞는지 검토하는 시간이 사라집니다. AI는 방대한 학습 데이터를 바탕으로 프레임워크의 관용적인 패턴을 이미 내재화하고 있기 때문입니다. 이 경험이 반복되면서 한 가지 생각이 점점 선명해졌습니다. 내가 프레임워크 API를 외워서 얻는 가치가 급격히 떨어지고 있다는 것. 이전에는 특정 프레임워크에 대한 깊은 지식이 곧 개발자의 경쟁력이라고 여겼는데, 그 전제가 흔들리고 있었습니다.

그런데 흥미로운 지점은 그 반대편에 있었습니다. AI가 놀라울 정도로 잘하는 영역이 있는 반면, 유독 힘을 쓰지 못하는 영역이 존재했습니다. 바로 **우리 비즈니스의 고유한 도메인 문제**입니다. 

* "태스크가 완료 상태이면 담당자를 재배정할 수 없다"는 규칙
* "VIP 고객의 환불 정책은 일반 고객과 다르게 적용된다"는 조건
* "이 워크플로우에서 승인 단계를 건너뛸 수 있는 사람은 팀장급 이상뿐이다"라는 제약

도메인의 고유한 문제일 수록 일반적인 AI가 알기 어렵습니다. 이것은 일반적인 지식이 아니라, 우리 팀이 오랜 시간 비즈니스를 운영하면서 쌓아 온 맥락이고, 수많은 논의와 실패를 통해 정제된 규칙이기 때문입니다. 아무리 뛰어난 AI라도 사전 학습 데이터에 없이는, 우리만의 문제를 대신 풀어줄 수는 없습니다. 도메인 지식은 여전히 철저하게 인간의 영역에 남아 있었습니다. 

이 두 가지 관찰이 겹치면서 하나의 결론에 도달했습니다. 개발자의 역할이 재정의되고 있다는 것입니다. 프레임워크 레이어의 작업, 즉 boilerplate 생성, API 연결, 데이터 접근 계층 구성 같은 일은 AI에게 위임할 수 있습니다. 점점 더 잘 위임할 수 있게 될 것입니다. 반면에 도메인 레이어의 작업, 즉 비즈니스 규칙을 이해하고, 올바르게 모델링하고, 그것이 정확히 동작하는지 검증하는 일은 개발자가 집중해야 할 본질적인 영역입니다. 

내가 해결해야 하는 고유한 문제에 시간을 써야 한다는 것. 이것이 AI 시대에 개발자가 가져야 할 방향이라는 확신이 생겼습니다. 이 인식이 DDD(Domain-Driven Design)라는 오래된 방법론과 만나면서, 구체적인 실천 방향이 보이기 시작했습니다.

## Spec DD > Test DD > Domain DD : AI 시대의 개발 방법론

앞서 두 가지 문제를 확인했습니다. 

* 프레임워크 중심 개발이 데이터 중심 설계로 이어진다는 것
* AI 시대에 개발자의 핵심 가치는 도메인에 있다는 것. 

이 두 인식이 만나는 지점에서, 구체적인 방법론이 형성됩니다. 

* SDD: Spec-Driven Development
* TDD: Test Driven Development
* DDD: Domain Driven Design

이것이 이 시리즈에서 제안하는 AI 시대의 개발 방법론입니다.

### Domain 중심 OOP — 비즈니스 로직을 도메인에 응집

데이터 중심 프로그래밍에서는 도메인 객체가 데이터 필드만 가진 빈 그릇이었습니다. 비즈니스 로직은 Service 레이어에 절차적으로 흩어져 있었고, 규칙이 어디에 있는지 파악하는 것 자체가 일이었습니다. DDD는 이 구조를 뒤집습니다. 도메인 객체가 데이터뿐 아니라 행위(behavior)를 함께 가지도록 만드는 것입니다. "태스크의 상태를 변경할 수 있는가?"라는 판단은 Service가 아니라 Task 객체 자신이 합니다. 비즈니스 로직이 도메인 객체 안에 응집되면, 코드를 수정할 위치가 자명해집니다. 응집도는 올라가고, 복잡도는 내려갑니다.

DDD에서는 이 도메인 모델을 구성하는 핵심 Building Block들이 있습니다.

- **Value Object**: 의미 있는 값을 표현하며, 불변이고, 동등성(equality)으로 비교합니다. 예를 들어 TaskTitle, Priority 같은 것들입니다.
- **Entity**: 고유한 ID를 가지며 생명주기를 가진 객체입니다. Task, Member 등이 여기에 해당합니다.
- **Aggregate**: 일관성(consistency)의 경계입니다. Aggregate 외부에서 내부 객체를 직접 수정할 수 없으며, 반드시 Aggregate Root를 통해야 합니다. Project가 대표적인 예입니다.
- **도메인 서비스**: 하나의 Entity에 귀속시키기 어려운, 여러 Entity에 걸친 비즈니스 로직을 담당합니다.
- **Repository**: 도메인 객체의 저장과 조회를 위한 인터페이스입니다. 인터페이스는 도메인 레이어에 정의하되, 실제 구현은 인프라 레이어에 둡니다.

이 Building Block들은 Part 2에서 하나씩 Spec 정의 -> TDD 테스트 작성 -> 구현 사이클로 만들어 갑니다.

### 프레임워크 Agnostic + POJO — TDD가 자연스러워지다

DDD의 도메인 모델은 순수한 TypeScript/JavaScript 객체, 즉 POJO(Plain Old JavaScript Object)로 구현합니다. NestJS의 데코레이터도, Spring의 어노테이션도, React의 훅도 필요 없습니다. 도메인 코드는 어떤 프레임워크에도 의존하지 않습니다.

이것이 왜 중요한가? 외부 의존성이 없는 순수 객체는 테스트하기 위해 Mock이 필요 없기 때문입니다. DB 연결을 흉내 낼 필요도 없고, HTTP 요청을 가로챌 필요도 없습니다. 그냥 객체를 생성하고, 메서드를 호출하고, 결과를 검증하면 됩니다. TDD가 자연스러워지는 것이 아니라, POJO 기반 도메인에서는 TDD를 하지 않을 이유가 없어집니다.

그리고 프레임워크가 바뀌어도 도메인 코드는 그대로입니다. Express에서 NestJS로 전환하든, REST에서 GraphQL로 바꾸든, 도메인 레이어는 영향을 받지 않습니다. 프레임워크는 도메인을 감싸는 얇은 어댑터일 뿐입니다. 도메인이 중심이고, 프레임워크가 주변부인 구조. 이것이 프레임워크 Agnostic 아키텍처의 핵심입니다.

### SDD (Spec-Driven Development) — AI와 함께 Spec 먼저

여기에 AI 시대의 새로운 요소를 하나 더합니다. SDD, Spec-Driven Development입니다.

SDD는 구현에 앞서 AI와의 대화를 통해 Spec을 먼저 정의하는 개발 방식입니다. BDD(Behavior-Driven Development)와 유사하게 행동 명세를 중심으로 개발하지만, 핵심적인 차이가 있습니다. Spec 정의의 대화 상대가 사람이 아니라 AI라는 점입니다. AI에게 도메인 규칙을 설명하고, AI가 이해한 바를 Spec 형태로 정리하고, 개발자가 이를 검증하고 보완합니다. 이 과정에서 도메인 지식이 명시적인 명세로 변환됩니다.

Spec이 정의되면 TDD 테스트 케이스는 자연스럽게 도출됩니다. Spec의 각 항목이 곧 테스트 케이스이기 때문입니다. 전체 워크플로우는 다음과 같습니다.

**(AI와) Spec 정의 -> TDD 테스트 작성 -> 도메인 구현 -> 프레임워크 연동 (AI 보조)**

이 워크플로우에서 주목할 점은, 인간의 영역과 AI의 영역이 자연스럽게 분리된다는 것입니다. 

* 도메인 지식을 바탕으로 Spec을 확정
* 그 Spec이 정확히 구현되었는지 TDD로 검증하기 위한 테스트케이스를 정의

개발자의 몫입니다. 반면 

* 테스트케이스가 확정된 이후 테스트 코드로 구현하는 것
* 테스트를 통화시키기 위하여 Spec 에 따라서 구현을 하는 것
* 도메인이 완성된 이후 프레임워크 연동, boilerplate 생성, 인프라 코드 작성

AI가 효율적으로 보조할 수 있는 영역입니다. 각자가 잘하는 일에 집중하는 구조입니다.
SDD의 실전 시연은 Ch.8에서 다룹니다.

### SDD+TDD+DDD — 이 시리즈가 말하고 싶은 것

정리하겠습니다. 
**SDD+TDD+DDD. 이것이 AI 시대의 개발 방법론입니다.**
 
SDD로 AI와 함께 Spec을 먼저 정의하고, TDD로 그 Spec을 테스트 케이스로 변환하여 검증하고, DDD로 비즈니스 로직을 도메인 객체에 응집시킵니다. 프레임워크는 도메인 바깥의 어댑터로 밀어내고, 그 영역은 AI에게 위임합니다. 개발자는 도메인에 집중합니다.

이 시리즈에서는 이 워크플로우를 태스크/프로젝트 관리 도메인을 통해 처음부터 끝까지 구현해 봅니다.
