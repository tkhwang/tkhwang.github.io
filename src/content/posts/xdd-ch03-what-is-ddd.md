이 글은 **[SDD+TDD+DDD] 시리즈**의 세 번째 글입니다. <br />

Ch.1에서는 왜 프레임워크 중심 사고에서 도메인 중심 사고로 무게중심을 옮겨야 하는지 이야기했습니다. Ch.2에서는 같은 요구사항을 데이터 위주와 객체 위주로 구현했을 때 코드가 어떻게 달라지는지 비교했습니다.

앞선 `Customer` 예제는 Value Object와 객체 위주 모델링의 차이를 빠르게 보여주기 위한 도입 예제였습니다. 이제부터는 더 복잡한 도메인을 다루기 전에, 먼저 DDD가 그 복잡성을 어떤 언어로 설명하는지부터 정리해 보겠습니다. 다만 DDD 이론을 한 글에 다 넣으면 오히려 숨이 가빠집니다. 그래서 이번 글과 다음 글, 두 편에 걸쳐 핵심 개념을 나누어 보겠습니다.

이번 글에서는 먼저 **전략적 설계의 입구**와 **가장 핵심적인 객체 구분**을 다룹니다. 즉 `Bounded Context`, `Ubiquitous Language`, `Value Object`, `Entity`, `Aggregate`를 먼저 붙잡겠습니다.

## 객체 위주 설계가 먼저다

DDD를 받아들이기 전에 먼저 필요한 것은 복잡한 이론이 아닙니다. 규칙이 붙은 문제를 데이터가 아니라 **값과 행위를 가지고 주체적으로 행동하는 객체로 다뤄야 한다는 감각**입니다.

예를 들어 이름, 이메일, 포인트 같은 값이 그냥 `string`, `number`로만 남아 있으면 규칙은 사용하는 곳마다 흩어집니다. 반대로 그 규칙을 객체 안으로 모으면, 코드는 "무엇을 지켜야 하는가"를 훨씬 분명하게 드러냅니다.

즉 순서는 이렇습니다.

- 먼저 객체 위주로 설계해야 할 필요를 느낀다
- 그다음 그 객체를 어떤 기준으로 나눌지 고민한다
- 그때 DDD가 강력한 설명 틀이 된다

## 이번 글의 범위

DDD는 크게 전략적 설계와 전술적 설계로 나눠 볼 수 있습니다.

- 전략적 설계는 도메인의 경계와 언어를 나누는 일이고
- 전술적 설계는 그 안에서 객체와 규칙을 코드로 모델링하는 일입니다

이번 글은 그중에서도 출발점에 해당하는 부분에 집중합니다. 먼저 전략적 설계에서 꼭 필요한 두 개념을 짚고, 그다음 전술적 설계의 가장 기본적인 Building Block인 `Value Object`, `Entity`, `Aggregate`를 설명하겠습니다.

## 전략적 설계에서 먼저 보는 두 가지

전략적 설계는 "도메인을 코드로 어떻게 짤 것인가"보다 먼저, **어디까지를 하나의 영역으로 볼 것인가**, 그리고 **그 영역 안에서 어떤 언어로 말할 것인가**를 정리하는 단계입니다. DDD에서 이 출발점을 잡아 주는 대표적인 개념이 `Bounded Context`와 `Ubiquitous Language`입니다.

- **Bounded Context**: 구분된 모듈이나 업무 영역의 경계
- **Ubiquitous Language**: Bounded Context 안에서 같은 대상을 같은 이름으로 부르는 공통 언어

이 둘이 중요한 이유는 먼저 **구분된 모듈이나 업무 영역의 경계**가 있어야 그 안에서 어떤 언어를 공통으로 쓸지도 정할 수 있기 때문입니다. 즉 `Bounded Context`는 "여기까지가 하나의 의미 체계"라고 선을 긋는 일이고, `Ubiquitous Language`는 그 경계 안에서 같은 대상을 모두 같은 이름으로 부르게 만드는 일입니다.

이 언어는 개발자들끼리만 맞추는 것이 아니라, 기획자나 도메인 전문가를 포함해 함께 일하는 모든 사람이 같은 개념을 같은 말로 이해하고 이야기하게 만드는 효과가 있습니다. 그래야 요구사항을 논의할 때와 설계를 할 때, Spec을 쓸 때와 테스트와 코드를 작성할 때의 의미가 서로 어긋나지 않습니다.

### 같은 단어가 컨텍스트마다 다를 수 있다

예를 들어 하나의 서비스 안에서 `Order`라는 단어를 여러 팀이 다르게 쓰는 상황을 생각해 봅시다.

- 주문 접수 팀은 `Order`를 "고객이 결제를 완료한 주문"이라는 뜻으로 씁니다
- 배송 팀은 `Order`를 "출고와 배송을 처리해야 하는 배송 단위"라는 뜻으로 씁니다
- 정산 팀은 `Order`를 "매출과 환불을 계산하는 정산 대상"이라는 뜻으로 씁니다

회의에서 "Order를 취소한다", "Order를 출고한다", "Order를 정산한다" 같은 말을 하면, 같은 단어를 쓰고 있어도 각자 떠올리는 대상이 달라질 수 있습니다.

이럴 때 먼저 해야 할 일은 **경계를 나누는 것**입니다.

- 주문 접수와 결제 상태를 다루는 영역
- 배송 준비와 출고를 다루는 영역
- 매출, 환불, 정산을 다루는 영역

이렇게 나누면 각 영역이 하나의 `Bounded Context`가 됩니다. 그다음에는 각 컨텍스트 안에서 공통 언어를 더 분명하게 정해야 합니다.

- 주문 컨텍스트에서는 `Order`를 "고객이 생성한 주문"으로 정의한다
- 배송 컨텍스트에서는 `Shipment`나 `FulfillmentOrder`처럼 배송 의미가 드러나는 이름을 쓴다
- 정산 컨텍스트에서는 `SettledOrder`나 `BillingRecord`처럼 정산 의미가 드러나는 이름을 쓴다

이렇게 하면 "같은 단어를 대충 공유하는 상태"에서 벗어나, **각 경계 안에서 어떤 단어가 무엇을 뜻하는지**가 분명해집니다. 이것이 `Ubiquitous Language`입니다.

## DDD는 "클래스를 많이 만드는 기법"이 아니다

DDD를 처음 접하면 쉽게 이렇게 오해합니다.

- 모든 것을 클래스화하는 방법론인가?
- Service를 Entity 안으로 옮기는 패턴인가?
- Java 스타일 OOP를 다시 하자는 이야기인가?

하지만 DDD의 핵심은 문법이나 프레임워크가 아닙니다. **도메인의 개념과 규칙을 코드 구조에 맞춰 담아내는 것**입니다. 즉 DDD는 "어떻게 클래스를 나눌까?"보다 먼저 이런 질문을 던집니다.

- 이 값은 그냥 데이터인가, 아니면 의미 있는 도메인 값인가?
- 이 대상은 식별자와 생명주기를 가져야 하는가?
- 어떤 규칙은 어디까지 하나의 경계 안에서 보장되어야 하는가?

이 질문에 답하다 보면, DDD의 핵심 Building Block이 자연스럽게 등장합니다.

## 먼저 두 종류로 나눠 생각하자

DDD 용어를 바로 외우기보다, 먼저 직관적으로 두 가지를 구분하면 이해가 쉽습니다.

- **규칙이 붙은 값**
- **시간에 따라 추적되는 주체**

예를 들어 고객 도메인을 생각해 보겠습니다.

- `name`은 비어 있을 수 없고, 길이 제한이 있고, 앞뒤 공백을 정리해야 합니다
- `email`은 형식 검증과 정규화 규칙이 있습니다
- `points`는 0 이상의 정수여야 하고, 적립과 차감 규칙이 붙습니다

이들은 모두 **규칙이 붙은 값**입니다. 반면 `customer`는 다릅니다.

- 고객마다 고유한 식별자가 있고
- 시간이 지나면서 정보가 바뀌고
- 시스템 안에서 계속 추적됩니다

이것은 **시간에 따라 추적되는 주체**입니다. DDD에서는 전자를 `Value Object`, 후자를 `Entity`라고 부릅니다.

## Value Object — 의미 있는 값을 표현하는 객체

Value Object는 단순한 값이 아닙니다. **도메인에서 의미를 가진 값**입니다. 예를 들면 이런 것들입니다.

- `Name`
- `Email`
- `RewardPoints`
- `ContentUrl`
- `Tag`

이 객체들의 공통점은 다음과 같습니다.

- 생성 규칙이 있다
- 불변 조건이 있다
- 보통 불변(immutable)으로 다룬다
- 식별자보다 값 자체가 중요하다

예를 들어 `Name("홍길동")`과 `Name("홍길동")`은 같은 값으로 봅니다. 별도의 id가 필요하지 않습니다. `Email`도 마찬가지입니다. `user@example.com`이라는 값 자체가 의미의 중심입니다.

그래서 Value Object는 대개 이런 역할을 맡습니다.

- 유효성 검증의 원본이 된다
- 정규화 정책을 한곳에 모은다
- 도메인 의미를 타입으로 드러낸다

Ch.2에서 `string` 대신 `Name`, `number` 대신 `RewardPoints`를 둔 이유가 바로 이것입니다. Value Object는 "기본형에 흩어진 규칙"을 객체 안으로 끌어옵니다.

물론 모든 값을 Value Object로 감쌀 필요는 없습니다. 비즈니스 규칙이 없고, 별도의 불변 조건이나 도메인 의미가 약한 값은 기본형으로 두는 편이 더 단순할 수 있습니다. 중요한 것은 기본형을 없애는 것이 아니라, **규칙이 붙은 값을 의미 있는 객체로 올려서 도메인 모델을 더 분명하게 만드는 것**입니다.

### Value Object는 DB에 어떻게 저장될까

여기서 자주 나오는 질문이 하나 있습니다. "Value Object는 개념은 알겠는데, 이건 DB에 어떻게 저장되나요? 이것도 DB Entity가 되나요?" 보통은 그렇지 않습니다. 일반적으로 Value Object는 독립적인 DB Entity가 되기보다, Domain Entity의 필드로 포함됩니다.

예를 들어 `Customer` Entity 안에 `Name`, `Email`, `RewardPoints`가 있다면, DB에 저장할 때는 결국 고객 레코드의 컬럼 값으로 풀려서 저장되는 경우가 많습니다. 즉 도메인 모델에서는 `Name`, `Email`, `RewardPoints`가 각각 의미 있는 객체이지만, persistence 관점에서는 보통 `customers.name`, `customers.email`, `customers.points` 같은 필드로 저장됩니다.

이 차이 때문에, 기본형만으로 설계한 경우에는 DB 모델과 도메인 모델이 거의 비슷한 모양이 되기 쉽습니다. 반면 Value Object를 도입하는 순간부터는 도메인 모델과 DB 모델이 조금씩 다른 형태와 역할을 가지기 시작합니다. 도메인 모델에서는 규칙과 의미를 가진 객체가 등장하고, DB 모델에서는 여전히 문자열과 숫자 컬럼으로 저장될 수 있습니다.

즉 Value Object는 **DB 중심 모델과 도메인 중심 모델이 분리되기 시작하는 출발점**이기도 합니다.

DDD를 처음 시작한다면, 가능한 한 기본형(`string`, `number`)을 그대로 쓰지 않는 습관을 들여 보시길 권합니다. 제약 조건이 있거나, 검증 규칙이 있거나, 비즈니스 의미가 붙는 값이라면 Value Object로 감싸 보는 것입니다. 처음에는 과하게 느껴질 수 있지만, 이 습관이 쌓이면 "규칙을 어디에 둘 것인가"를 자연스럽게 객체 안에서 먼저 생각하게 됩니다. 그것이 객체 중심 설계에 익숙해지는 가장 실질적인 첫걸음입니다.

## Entity — 식별자와 생명주기를 가진 객체

Value Object가 규칙이 붙은 "값"이라면, 그와는 다른 부류도 있습니다. 값처럼 그때그때 교체되는 것이 아니라, 시스템 안에서 오랫동안 유지되고 추적되는 대상입니다. 시간이 지나면서 상태가 바뀌어도 같은 것으로 취급되고, 생성부터 변경, 종료까지의 흐름을 가집니다.

DDD에서는 이런 대상을 **Entity**로 모델링합니다. Entity는 값이 아니라 **정체성(identity)** 이 중심인 객체입니다.
예를 들어 `Customer`를 생각해 봅시다.

- 이름이나 이메일이 바뀌어도 같은 고객일 수 있습니다
- 포인트가 1000에서 1200으로 바뀌어도 같은 고객입니다
- 중요한 것은 "이 객체가 누구인가"이지, "지금 값이 정확히 무엇인가"만은 아닙니다

데이터 위주 모델링에 익숙하다면, Entity를 처음에는 DB Entity와 비슷하게 느낄 수 있습니다. 실제로 둘 다 식별자를 가지고, 시간이 지나도 같은 대상으로 취급된다는 점에서는 닮아 있습니다. 하지만 DDD의 Entity는 테이블 구조보다 **정체성과 생명주기, 그리고 상태 변경의 규칙**이 더 중요합니다.

이 차이는 전략적 설계 관점에서도 볼 수 있습니다. 둘 다 `Entity`라는 같은 단어를 쓰고 있지만, 사실은 서로 다른 컨텍스트 안에서 사용되고 있습니다. DB Entity는 저장과 조회를 중심으로 보는 인프라/데이터베이스 컨텍스트의 언어이고, Domain Entity는 비즈니스 규칙과 생명주기를 중심으로 보는 도메인 컨텍스트의 언어입니다.

이 시리즈에서 `Entity`라고 하면 **DB 테이블 구조가 아니라 도메인 안에서 식별되고 추적되는 주체**를 뜻한다고 생각하시면 됩니다.

이런 대상은 고유한 식별자를 가집니다.

- `CustomerId`
- `ContentId`
- `UserId`

그리고 시간에 따라 상태가 변합니다.

- 고객 정보가 변경된다
- 콘텐츠의 소비 상태가 바뀐다
- 북마크한 콘텐츠가 소비 완료 상태로 전이된다

즉 Entity는 "데이터를 담는 레코드"가 아니라, **자기 생명주기와 규칙을 가진 주체**입니다.

### Entity는 DB에 어떻게 저장될까

Domain Entity는 보통 DB에서 **하나의 테이블(또는 row)** 에 대응됩니다. 예를 들어 `Customer` Entity는 `customers` 테이블의 한 행으로 저장되고, `Content` Entity는 `contents` 테이블의 한 행으로 저장됩니다.

겉보기에는 DB Entity와 비슷해 보이지만, 역할이 다릅니다. DB Entity는 컬럼과 관계로 **데이터를 저장하는 구조** 입니다. 데이터만 있고, 그 데이터를 바꾸는 것은 바깥의 Service나 쿼리가 합니다.

반면 Domain Entity는 데이터와 함께 **행위(method)** 를 가집니다. 예를 들어 `content.complete()`처럼 객체가 스스로 자기 상태를 바꿉니다. 외부에서 `content.status = 'Done'`으로 직접 값을 넣는 것이 아니라, 객체가 주체적으로 규칙을 지키면서 상태를 전이합니다. 이렇게 하면 상태 변경에 필요한 규칙이 객체 안에 응집력 있게 모이고, 코드도 "무엇을 하는가"가 더 분명하게 읽힙니다.

저장 관점에서 보면, Domain Entity의 필드들은 도메인 안에서는 `Name`, `Email`, `RewardPoints` 같은 Value Object로 표현됩니다. 하지만 DB에 저장할 때 Value Object가 별도 테이블로 나뉘는 것은 아닙니다. 이 Value Object들은 다시 기본형(`string`, `number`)으로 풀려서, Entity의 테이블 컬럼에 포함되어 저장됩니다. 즉 도메인에서는 의미 있는 객체로 다루지만, DB에서는 Entity의 일부로 함께 저장되는 것입니다. 이 변환은 뒤에서 설명할 Mapper가 맡습니다.

## Aggregate — 규칙을 함께 지켜야 하는 경계

Value Object와 Entity까지는 비교적 직관적입니다. 많은 사람이 Aggregate에서 처음 막힙니다. Aggregate를 가장 간단하게 말하면 **일관성을 보장하는 경계**입니다.

하나의 Aggregate는 보통 Aggregate Root를 중심으로, 함께 동작하는 여러 Entity와 Value Object로 구성됩니다. 중요한 것은 객체를 많이 묶는 것이 아니라, **어떤 규칙을 하나의 변경 단위 안에서 함께 지켜야 하는지를 정하는 것**입니다.

이렇게 보면 Aggregate는 단순히 Entity를 하나 더 감싼 구조라기보다, **Entity 개념이 더 큰 일관성 경계로 확장된 것**에 가깝습니다. Entity가 "이 객체가 스스로 어떤 규칙을 지켜야 하는가"에 집중한다면, Aggregate는 "이 객체와 그 내부 구성 요소들이 함께 어떤 규칙을 지켜야 하는가"를 다룹니다.

### Aggregate를 어려워하는 이유

Aggregate에서 특히 많이 막히는 이유는, 처음부터 도메인보다 **DB 테이블 구조**를 먼저 떠올리기 때문이기도 합니다. 테이블 중심으로 생각하면 `Content`, `Rating`, `Tag`를 각각 별도 테이블이나 관계로 나누는 데는 익숙하지만, "이 객체들이 어떤 규칙 아래 함께 움직여야 하는가"를 한 덩어리로 보는 감각은 잘 생기지 않습니다.

그래서 Aggregate를 설계할 때는 **Domain First, Persistence Later**라는 태도가 꽤 도움이 됩니다. 처음부터 "이걸 어떤 테이블에 저장하지?"를 고민하기보다, 우선 도메인 객체를 먼저 떠올리고 어떤 규칙이 하나의 경계 안에서 함께 지켜져야 하는지를 정합니다. 그리고 그 도메인 객체가 충분히 정리된 뒤에, 그다음 단계에서 persistence를 위한 Repository와 DB 매핑을 도입하는 편이 훨씬 자연스럽습니다.

### Aggregate는 transaction 단위와도 자주 연결된다

이 개념은 persistence 관점에서 보면, 보통 **한 번에 원자적으로(atomic) 처리되어야 하는 DB transaction의 단위**와도 자주 연결됩니다. 도메인에서는 "이 규칙들이 함께 지켜져야 하는 경계"로 보지만, 저장 단계에서는 그 경계 안의 변경이 한 트랜잭션 안에서 함께 성공하거나 함께 실패해야 하는 경우가 많기 때문입니다.

즉 Aggregate는 단순한 객체 묶음이 아니라, 도메인에서는 **일관성 경계**이고 DB 관점에서는 자주 **하나의 transaction 단위**로 대응되는 개념입니다.

### 예시로 보면 더 쉽다

이후 시리즈에서 본격적으로 다룰 콘텐츠 북마크와 소비 추적 도구 도메인을 미리 맛보기로 살펴보겠습니다.

- `Content`는 소비 상태(`Bookmarked -> InProgress -> Done`)를 가진다
- 완료되지 않은 콘텐츠에는 `Rating`을 줄 수 없다
- 상태 전이와 평점 부여 규칙은 함께 일관되게 지켜져야 한다

이런 규칙은 개별 값 하나만 보고는 지킬 수 없습니다. `Content` 전체를 함께 봐야 합니다. 이럴 때 `Content`를 Aggregate Root로 두고, 상태 전이와 평점 부여 같은 규칙을 그 경계 안에서 관리합니다.

핵심은 이것입니다.

- Aggregate 밖에서는 내부 객체를 마음대로 바꾸지 않는다
- 상태 변경은 Aggregate Root를 통해서만 일어난다
- 그 경계 안에서 필요한 불변식을 함께 지킨다

즉 Aggregate는 "객체를 묶는 컨테이너"가 아니라, **규칙을 같이 지켜야 하는 단위**입니다.

### Aggregate는 DB에 어떻게 저장될까

Aggregate는 하나의 테이블에 1:1로 대응되지 않는 경우가 많습니다. 예를 들어 `Content` Aggregate 안에 `ContentUrl`, `Tag`, `Rating` 같은 Value Object가 있다면, 저장 방식은 여러 가지입니다.

- `ContentUrl`이나 `Rating`처럼 단일 값은 `contents` 테이블의 컬럼으로 저장될 수 있다
- `Tag`처럼 여러 개인 값은 별도 테이블(`content_tags`)로 나뉘거나, JSON 컬럼으로 저장될 수 있다

중요한 것은 이런 저장 구조를 먼저 정하는 것이 아니라, **도메인 모델을 먼저 설계하고 나서 저장 방식을 결정한다**는 점입니다. 앞서 말한 "Domain First, Persistence Later"가 여기서도 적용됩니다. 그리고 이 도메인 모델과 DB 구조 사이의 변환은 Repository와 Mapper가 담당합니다.

### 정리: 개념과 코드의 대응

여기까지 Value Object, Entity, Aggregate를 각각 설명하고 DB 저장 방식도 살펴봤습니다. 마지막으로 헷갈리기 쉬운 대응 관계를 개념 수준에서만 한 번 정리해 보겠습니다.

| 개념 | 클래스 | 역할 | 질문 |
|---|---|---|---|
| **Aggregate Root** | 어떤 Aggregate의 중심 Entity | 일관성 경계의 진입점, 외부 접근의 창구 | "이 경계 안에서 누가 중심인가?" |
| **Entity** | 식별자와 생명주기를 가진 객체 | 시간에 따라 추적되는 주체 | "이 객체는 시간에 따라 추적되는가?" |
| **Value Object** | 규칙이 붙은 값 객체 | 의미와 제약을 가진 값 | "이 값은 의미와 제약을 가지는가?" |

Aggregate는 이 표에 없습니다. Aggregate는 별도 클래스가 아니라, Aggregate Root를 중심으로 그 안의 Entity와 Value Object들까지 포함하는 **개념적 경계** 전체를 가리킵니다.

Aggregate 안에 Entity가 하나뿐일 때는 Aggregate Root와 Entity가 같은 객체가 될 수 있습니다. Aggregate가 복잡해지면 내부에 하위 Entity가 더 생길 수 있습니다.

실제 도메인에서 어떤 객체가 Aggregate Root이고, 어떤 값들이 Value Object로 붙는지는 요구사항을 분해해 봐야 보입니다. 그 실제 대응은 다음 글들에서 본편 도메인으로 옮겨 가며 살펴보겠습니다.

## 여기까지가 DDD 첫 번째 입니다.

정리하면 이번 글에서 소개한 DDD의 첫 번째 핵심 어휘는 다음과 같습니다.

- **Bounded Context**: 구분된 업무 영역의 경계
- **Ubiquitous Language**: 그 경계 안에서 공유하는 공통 언어
- **Value Object**: 의미 있는 값을 표현하는 객체
- **Entity**: 식별자와 생명주기를 가진 객체
- **Aggregate**: 규칙을 함께 지켜야 하는 일관성 경계

이제 중요한 것은 용어를 외우는 것이 아닙니다. **요구사항을 읽었을 때 어떤 것이 값이고, 어떤 것이 주체인지 판단할 수 있는가**입니다.
