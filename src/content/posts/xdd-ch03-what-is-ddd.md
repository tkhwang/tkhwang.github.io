이 글은 **[SDD+TDD+DDD] 시리즈**의 세 번째 글입니다. <br />

Ch.1에서는 왜 프레임워크 중심 사고에서 도메인 중심 사고로 무게중심을 옮겨야 하는지 이야기했습니다. Ch.2에서는 같은 요구사항을 데이터 위주와 객체 위주로 구현했을 때 코드가 어떻게 달라지는지 비교했습니다.

앞선 Customer 예제는 Value Object와 객체 위주 모델링의 차이를 빠르게 보여주기 위한 도입 예제였습니다. 이제부터는 더 복잡한 도메인을 다루기 전에, 먼저 DDD가 그 복잡성을 어떤 언어로 설명하는지부터 정리해 보겠습니다. 여기까지 왔으면 먼저 한 가지는 받아들인 셈입니다. 비즈니스 규칙이 중요한 소프트웨어라면, 데이터를 여기저기서 절차적으로 만지는 것보다 **객체 위주로 설계하는 편이 더 자연스럽다**는 점입니다.

그런데 객체 위주 설계를 하려고 하면 곧바로 다음 질문이 생깁니다. 기존 데이터 중심 설계에서는 대개 DB 스키마를 먼저 떠올리고, 컬럼 타입에 맞춰 `string`, `number`, `boolean`을 배치하는 데서 출발했습니다. 그 방식에서는 "무엇이 값이고 무엇이 주체인가"를 깊게 묻지 않아도 일단 코드를 만들 수 있습니다. 반면 객체 위주 설계는 질문이 달라집니다. 어떤 것은 단순한 값이 아니라 규칙을 가진 객체처럼 다뤄야 하고, 어떤 것은 식별자와 생명주기를 가진 주체처럼 다뤄야 합니다. **그 객체를 어떤 기준으로 나눌 것인가?**

이 지점에서 DDD가 좋은 언어가 됩니다. DDD는 객체 위주 설계를 더 선명하게 설명하고, 도메인 규칙을 어떤 구조로 담을지 판단하게 해 주는 어휘를 제공합니다. 이번 글은 바로 그 지점을 정리합니다. 다만 이 글의 목표는 DDD 전체를 완결적으로 해설하는 것이 아닙니다. **이 시리즈를 따라가기 위해 필요한 만큼의 DDD 기본 개념을 먼저 잡는 것**이 목적입니다. 뒤로 가면서 Aggregate, Domain Service, Repository 같은 주제는 실제 설계와 구현 맥락 안에서 더 자세히 다시 다루게 됩니다.

## 객체 위주 설계가 먼저다

DDD를 받아들이기 전에 먼저 필요한 것은 복잡한 이론이 아닙니다. 규칙이 붙은 문제를 데이터가 아니라 **값과 행위를 가지고 주체적으로 행동하는 객체로 다뤄야 한다는 감각**입니다. 예를 들어 이름, 이메일, 포인트 같은 값이 그냥 `string`, `number`로만 남아 있으면 규칙은 사용하는 곳마다 흩어집니다. 반대로 그 규칙을 객체 안으로 모으면, 코드는 "무엇을 지켜야 하는가"를 훨씬 분명하게 드러냅니다.

즉 순서는 이렇습니다.

- 먼저 객체 위주로 설계해야 할 필요를 느낀다
- 그다음 그 객체를 어떤 기준으로 나눌지 고민한다
- 그때 DDD가 강력한 설명 틀이 된다

## 이번 글의 범위

DDD는 크게 전략적 설계와 전술적 설계로 나눠 볼 수 있습니다.

* 전략적 설계는 도메인의 경계와 언어를 나누는 일이고
* 전술적 설계는 그 안에서 객체와 규칙을 코드로 모델링하는 일입니다.

이번 글은 후자, 즉 **전술적 설계의 출발점**에 집중합니다.<br />다만 그 전에 전략적 설계 중에서 다음 두 가지는 짧게 짚고 가겠습니다.

### 전략적 설계에서 먼저 보는 두 가지

전략적 설계는 "도메인을 코드로 어떻게 짤 것인가"보다 먼저, **어디까지를 하나의 영역으로 볼 것인가**, 그리고 **그 영역 안에서 어떤 언어로 말할 것인가**를 정리하는 단계입니다. DDD에서 이 출발점을 잡아 주는 대표적인 개념이 `Bounded Context`와 `Ubiquitous Language`입니다.

- **Bounded Context**: 구분된 모듈이나 업무 영역의 경계
- **Ubiquitous Language**: Bounded Context 안에서 같은 대상을 같은 이름으로 부르는 공통 언어

이 둘이 중요한 이유는, 먼저 **구분된 모듈이나 업무 영역의 경계**가 있어야 그 안에서 어떤 언어를 공통으로 쓸지도 정할 수 있기 때문입니다. 즉 Bounded Context는 "여기까지가 하나의 의미 체계"라고 선을 긋는 일이고, Ubiquitous Language는 그 경계 안에서 같은 대상을 모두 같은 이름으로 부르게 만드는 일입니다.

이 언어는 개발자들끼리만 맞추는 것이 아니라, 기획자나 도메인 전문가를 포함해 함께 일하는 모든 사람이 같은 개념을 같은 말로 이해하고 이야기하게 만드는 효과가 있습니다. 그래야 요구사항을 논의할 때와 설계를 할 때, Spec을 쓸 때와 테스트와 코드를 작성할 때의 의미가 서로 어긋나지 않습니다. 같은 단어라도 영역이 바뀌면 의미가 달라질 수 있기 때문에, 설계와 Spec, 테스트, 코드는 항상 "어느 컨텍스트 안의 언어인가"를 의식해야 합니다.

예를 들어 하나의 서비스 안에서 `Order`라는 단어를 여러 팀이 다르게 쓰는 상황을 생각해 봅시다.

- 주문 접수 팀은 `Order`를 "고객이 결제를 완료한 주문"이라는 뜻으로 씁니다.
- 배송 팀은 `Order`를 "출고와 배송을 처리해야 하는 배송 단위"라는 뜻으로 씁니다.
- 정산 팀은 `Order`를 "매출과 환불을 계산하는 정산 대상"이라는 뜻으로 씁니다.

회의에서 "Order를 취소한다", "Order를 출고한다", "Order를 정산한다" 같은 말을 하면, 같은 단어를 쓰고 있어도 각자 떠올리는 대상이 달라질 수 있습니다. 누군가는 주문 접수 상태를 생각하고, 누군가는 배송 지시 단위를 생각하고, 누군가는 매출 계산 대상을 떠올릴 수 있습니다.

이럴 때 먼저 해야 할 일은 **경계를 나누는 것**입니다.

- 주문 접수와 결제 상태를 다루는 영역
- 배송 준비와 출고를 다루는 영역
- 매출, 환불, 정산을 다루는 영역

이렇게 나누면 각 영역이 하나의 `Bounded Context`가 됩니다. 즉 "여기서는 어떤 의미 체계가 유효한가"를 먼저 정하는 것입니다.
그다음에는 각 컨텍스트 안에서 **공통 언어를 명확히 정해야 합니다.**

- 주문 컨텍스트에서는 `Order`를 "고객이 생성한 주문"으로 정의한다
- 배송 컨텍스트에서는 `Shipment`나 `FulfillmentOrder`처럼 배송 의미가 드러나는 이름을 쓴다
- 정산 컨텍스트에서는 `SettledOrder`나 `BillingRecord`처럼 정산 의미가 드러나는 이름을 쓴다

이렇게 하면 "같은 단어를 대충 공유하는 상태"에서 벗어나, **각 경계 안에서 어떤 단어가 무엇을 뜻하는지**가 분명해집니다. 이것이 `Ubiquitous Language`입니다. 전략적 설계는 바로 이런 혼선을 줄이기 위해, 먼저 경계를 나누고 그 경계 안의 언어를 맞추는 작업입니다. 이 시리즈에서는 우선 하나의 비교적 단순한 컨텍스트 안에서 전술적 설계를 살펴보겠습니다.

그 위에서 이번 글에서는 실제 구현에 중요한 전술적 설계 중에서 다음 일곱 가지만 먼저 붙잡겠습니다.

- Value Object
- Entity
- Aggregate
- Domain Service
- Repository
- Mapper
- Domain Event

이 일곱 가지는 이후 시리즈를 따라가면서 설계, Spec, 구현, 저장, 모듈 간 협업을 이해할 때 계속 등장합니다. 다만 Ch.4에서는 우선 Value Object와 Entity 후보를 뽑는 데 집중하고, Mapper나 Domain Event 같은 주제는 뒤에서 더 자세히 다루는 편이 자연스럽습니다.

## DDD는 "클래스를 많이 만드는 기법"이 아니다

DDD를 처음 접하면 쉽게 이렇게 오해합니다.

- 모든 것을 클래스화하는 방법론인가?
- Service를 Entity 안으로 옮기는 패턴인가?
- Java 스타일 OOP를 다시 하자는 이야기인가?

하지만 DDD의 핵심은 문법이나 프레임워크가 아닙니다. **도메인의 개념과 규칙을 코드 구조에 맞춰 담아내는 것** 입니다.
즉 DDD는 "어떻게 클래스를 나눌까?"보다 먼저, 이런 질문을 던집니다.

- 이 값은 그냥 데이터인가, 아니면 의미 있는 도메인 값인가?
- 이 대상은 식별자와 생명주기를 가져야 하는가?
- 어떤 규칙은 어디까지 하나의 경계 안에서 보장되어야 하는가?

이 질문에 답하다 보면, DDD의 핵심 Building Block이 자연스럽게 등장합니다.

## 먼저 두 종류로 나눠 생각하자

DDD 용어를 바로 외우기보다, 먼저 직관적으로 두 가지를 구분하면 이해가 쉽습니다.

- **규칙이 붙은 값**
- **시간에 따라 추적되는 주체**

예를 들어 고객 도메인을 생각해 보겠습니다.

- `name`은 비어 있을 수 없고, 길이 제한이 있고, 앞뒤 공백을 정리해야 합니다.
- `email`은 형식 검증과 정규화 규칙이 있습니다.
- `points`는 0 이상의 정수여야 하고, 적립과 차감 규칙이 붙습니다.

이들은 모두 **규칙이 붙은 값**입니다.
반면 `customer`는 다릅니다.

- 고객마다 고유한 식별자가 있고
- 시간이 지나면서 정보가 바뀌고
- 시스템 안에서 계속 추적됩니다

이것은 **시간에 따라 추적되는 주체**입니다.
DDD에서는 전자를 `Value Object`, 후자를 `Entity`라고 부릅니다.

## Value Object — 의미 있는 값을 표현하는 객체

Value Object는 단순한 값이 아닙니다. **도메인에서 의미를 가진 값**입니다. 예를 들면 이런 것들입니다.

- `Name`
- `Email`
- `RewardPoints`
- `ContentUrl`
- `Tag`

이 객체들의 공통점은 다음과 같습니다.

- 생성 규칙이 있다
- 불변 조건이 있다
- 보통 불변(immutable)으로 다룬다
- 식별자보다 값 자체가 중요하다

예를 들어 `Name("홍길동")`과 `Name("홍길동")`은 같은 값으로 봅니다. 별도의 id가 필요하지 않습니다. `Email`도 마찬가지입니다. `user@example.com`이라는 값 자체가 의미의 중심입니다. 그래서 Value Object는 대개 이런 역할을 맡습니다.

- 유효성 검증의 원본이 된다
- 정규화 정책을 한곳에 모은다
- 도메인 의미를 타입으로 드러낸다

Ch.2에서 `string` 대신 `Name`, `number` 대신 `RewardPoints`를 둔 이유가 바로 이것입니다. Value Object는 "기본형에 흩어진 규칙"을 객체 안으로 끌어옵니다.

물론 모든 값을 Value Object로 감쌀 필요는 없습니다. 비즈니스 규칙이 없고, 별도의 불변 조건이나 도메인 의미가 약한 값은 기본형으로 두는 편이 더 단순할 수 있습니다. 중요한 것은 기본형을 없애는 것이 아니라, **규칙이 붙은 값을 의미 있는 객체로 올려서 도메인 모델을 더 분명하게 만드는 것**입니다.

여기서 자주 나오는 질문이 하나 있습니다. "Value Object는 개념은 알겠는데, 이건 DB에 어떻게 저장되나요? 이것도 DB 엔터티가 되나요?" 보통은 그렇지 않습니다. 일반적으로 Value Object는 독립적인 DB 엔터티가 되기보다, 도메인 Entity의 필드로 포함됩니다. 예를 들어 `Customer` Entity 안에 `Name`, `Email`, `RewardPoints`가 있다면, DB에 저장할 때는 결국 고객 레코드의 컬럼 값으로 풀려서 저장되는 경우가 많습니다.

즉 도메인 모델에서는 `Name`, `Email`, `RewardPoints`가 각각 의미 있는 객체이지만, persistence 관점에서는 보통 `customers.name`, `customers.email`, `customers.points` 같은 필드로 저장됩니다. 중요한 것은 저장 구조보다 도메인 모델 안에서 어떤 규칙을 어디에 둘 것인가입니다. Value Object를 먼저 도메인 개념으로 이해하고, 저장은 그다음에 생각하는 편이 자연스럽습니다.

이 차이 때문에, 기본형만으로 설계한 경우에는 DB 모델과 도메인 모델이 거의 비슷한 모양이 되기 쉽습니다. `name: string`, `email: string`, `points: number`처럼 컬럼 구조와 코드 구조가 1:1로 대응하기 때문입니다. 반면 Value Object를 도입하는 순간부터는 도메인 모델과 DB 모델이 조금씩 다른 형태와 역할을 가지기 시작합니다. 도메인 모델에서는 `Name`, `Email`, `RewardPoints`처럼 규칙과 의미를 가진 객체가 등장하고, DB 모델에서는 여전히 그것이 문자열과 숫자 컬럼으로 저장될 수 있습니다.

즉 Value Object는 단지 "클래스를 하나 더 만드는 것"이 아니라, **DB 중심 모델과 도메인 중심 모델이 분리되기 시작하는 출발점**이기도 합니다. 이 분리는 매핑이라는 추가 작업을 만들지만, 그 대신 도메인 규칙이 더 분명하게 코드 안에 자리 잡게 해 줍니다.

DB 모델이 대체로 "어떤 데이터를 저장할 것인가"에 초점이 맞춰져 있다면, 도메인 모델은 "이 데이터가 어떤 규칙 아래 어떻게 동작해야 하는가"까지 함께 표현합니다. DB 모델에는 보통 데이터 필드만 있지만, 도메인 모델은 데이터와 함께 행위(method)를 가지면서 스스로 규칙을 지키고 판단하는 **주체적인 객체**로 동작할 수 있게 됩니다. 이 차이가 쌓일수록 도메인 로직은 DB 구조보다 도메인 모델 쪽에 더 자연스럽게 응집됩니다.

이런 차이 때문에, 도메인 레이어에서 DB 레이어로 내려가 저장할 때나 반대로 DB 데이터를 읽어 도메인 모델로 다시 만들 때는 **서로 다른 형태를 변환하는 작업**이 필요합니다. 예를 들어 DB에서는 `name`, `email`, `points`가 기본형 컬럼으로 저장되어 있지만, 도메인 모델에서는 이를 `Name`, `Email`, `RewardPoints`로 다시 감싸서 의미 있는 객체로 복원해야 합니다.

DDD에서는 이런 역할을 흔히 `Mapper`가 담당합니다. Mapper는 DB 모델을 도메인 모델로 바꾸고, 도메인 모델을 다시 저장 가능한 형태로 바꾸는 변환 책임을 맡습니다. 처음 DDD를 접할 때 많은 분이 여기서 한 번 더 어려움을 느낍니다. "왜 이렇게 한 번 더 바꿔야 하지?"라는 질문이 자연스럽기 때문입니다. 하지만 이 변환은, 저장 구조와 도메인 규칙의 책임을 분리하기 위해 필요한 비용이라고 볼 수 있습니다. Mapper와 매핑 전략은 뒤에서 Repository와 함께 조금 더 자세히 다루겠습니다.

## Entity — 식별자와 생명주기를 가진 객체

Value Object가 규칙이 붙은 "값"이라면, 그와는 다른 부류도 있습니다. 값처럼 그때그때 교체되는 것이 아니라, 시스템 안에서 오랫동안 유지되고 추적되는 대상입니다. 시간이 지나면서 상태가 바뀌어도 같은 것으로 취급되고, 생성부터 변경, 종료까지의 흐름을 가집니다.

DDD에서는 이런 대상을 **Entity**로 모델링합니다. Entity는 값이 아니라 **정체성(identity)** 이 중심인 객체입니다.
예를 들어 `Customer`를 생각해 봅시다.

- 이름이나 이메일이 바뀌어도 같은 고객일 수 있습니다.
- 포인트가 1000에서 1200으로 바뀌어도 같은 고객입니다.
- 중요한 것은 "이 객체가 누구인가"이지, "지금 값이 정확히 무엇인가"만은 아닙니다.

데이터 위주 모델링에 익숙하다면, Entity를 처음에는 DB 엔터티와 비슷하게 느낄 수 있습니다. 실제로 둘 다 식별자를 가지고, 시간이 지나도 같은 대상으로 취급된다는 점에서는 닮아 있습니다. 하지만 DDD의 Entity는 테이블 구조보다 **정체성과 생명주기, 그리고 상태 변경의 규칙**이 더 중요합니다. 이 부분은 용어 때문에 특히 헷갈리기 쉽습니다. 데이터베이스에서 말하는 **DB 엔터티**와 DDD에서 말하는 `Entity`는 완전히 같은 개념이 아닙니다. DB 엔터티는 테이블의 row에 대응하는 데이터 구조에 가깝지만, DDD의 Entity는 비즈니스 규칙과 상태 전이를 스스로 책임지는 주체입니다.

이 차이는 전략적 설계 관점에서도 볼 수 있습니다. 둘 다 `Entity`라는 같은 단어를 쓰고 있지만, 사실은 서로 다른 컨텍스트 안에서 사용되고 있습니다. DB 엔터티는 저장과 조회를 중심으로 보는 인프라/데이터베이스 컨텍스트의 언어이고, Domain Entity는 비즈니스 규칙과 생명주기를 중심으로 보는 도메인 컨텍스트의 언어입니다. 즉 이름은 같아도, **어느 Bounded Context 안에서 쓰는 말인가에 따라 의미가 달라집니다.**

이 시리즈에서 `Entity`라고 하면 **DB 테이블 구조가 아니라 도메인 안에서 식별되고 추적되는 주체**를 뜻한다고 생각하시면 됩니다.
이런 대상은 고유한 식별자를 가집니다.

- `CustomerId`
- `ContentId`
- `UserId`

그리고 시간에 따라 상태가 변합니다.

- 고객 정보가 변경된다
- 콘텐츠의 소비 상태가 바뀐다
- 북마크한 콘텐츠가 소비 완료 상태로 전이된다

그래서 Entity는 보통 다음 질문과 함께 다뤄집니다.

- 어떤 상태 변경을 허용할 것인가?
- 어떤 순서의 상태 전이가 가능한가?
- 어떤 규칙을 스스로 지켜야 하는가?

즉 Entity는 "데이터를 담는 레코드"가 아니라, **자기 생명주기와 규칙을 가진 주체**입니다.

## Aggregate — 규칙을 함께 지켜야 하는 경계

Value Object와 Entity까지는 비교적 직관적입니다. 많은 사람이 Aggregate에서 처음 막힙니다. Aggregate를 가장 간단하게 말하면 **일관성을 보장하는 경계**입니다. 하나의 Aggregate는 보통 Aggregate Root를 중심으로, 함께 동작하는 여러 Entity와 Value Object로 구성됩니다. 중요한 것은 객체를 많이 묶는 것이 아니라, 어떤 규칙을 하나의 변경 단위 안에서 함께 지켜야 하는지를 정하는 것입니다. 이런 이유로 Aggregate는 데이터 위주 모델링에 익숙한 관점에서 보면, 보통 하나의 DB 트랜잭션 안에서 규칙을 함께 지켜야 하는 단위로 이해할 수 있습니다.

이렇게 보면 Aggregate는 단순히 Entity를 하나 더 감싼 구조라기보다, **Entity 개념이 더 큰 일관성 경계로 확장된 것**에 가깝습니다. Entity가 "이 객체가 스스로 어떤 규칙을 지켜야 하는가"에 집중한다면, Aggregate는 "이 객체와 그 내부 구성 요소들이 함께 어떤 규칙을 지켜야 하는가"를 다룹니다. 즉 Aggregate는 개별 Entity의 생명주기와 규칙을 넘어서, 여러 객체를 아우르는 일관성의 책임까지 끌어안는 단위입니다.

이 개념은 persistence 관점에서 보면, 보통 **한 번에 원자적으로(atomic) 처리되어야 하는 DB transaction의 단위**와도 자주 연결됩니다. 도메인에서는 "이 규칙들이 함께 지켜져야 하는 경계"로 보지만, 저장 단계에서는 그 경계 안의 변경이 한 트랜잭션 안에서 함께 성공하거나 함께 실패해야 하는 경우가 많기 때문입니다. 즉 Aggregate는 단순한 객체 묶음이 아니라, 도메인에서는 일관성 경계이고 DB 관점에서는 자주 하나의 transaction 단위로 대응되는 개념입니다.

Aggregate에서 특히 많이 막히는 이유는, 처음부터 도메인보다 **DB 테이블 구조**를 먼저 떠올리기 때문이기도 합니다. 테이블 중심으로 생각하면 `Content`, `Rating`, `Tag`를 각각 별도 테이블이나 관계로 나누는 데는 익숙하지만, "이 객체들이 어떤 규칙 아래 함께 움직여야 하는가"를 한 덩어리로 보는 감각은 잘 생기지 않습니다. DB에서는 분리된 row와 relation이 먼저 보이지만, 도메인에서는 그보다 **함께 지켜야 하는 규칙과 상태 변경의 단위**가 먼저 보여야 합니다.

그래서 Aggregate를 설계할 때는 **Domain First, Persistence Later**라는 태도가 꽤 도움이 됩니다. 처음부터 "이걸 어떤 테이블에 저장하지?", "외래 키는 어떻게 두지?"를 고민하기보다, 우선 도메인 객체를 먼저 떠올리고 어떤 규칙이 하나의 경계 안에서 함께 지켜져야 하는지를 정합니다. 그리고 그 도메인 객체가 충분히 정리된 뒤에, 그다음 단계에서 persistence를 위한 Repository와 DB 매핑을 도입하는 편이 훨씬 자연스럽습니다.

예를 들어 이후 시리즈에서 본격적으로 다룰 콘텐츠 북마크 & 소비 추적 도구 도메인을 미리 맛보기로 살펴보겠습니다.

- `Content`는 소비 상태(`Bookmarked -> InProgress -> Done`)를 가진다
- 완료되지 않은 콘텐츠에는 `Rating`을 줄 수 없다
- 상태 전이와 평점 부여 규칙은 함께 일관되게 지켜져야 한다

이런 규칙은 개별 값 하나만 보고는 지킬 수 없습니다. `Content` 전체를 함께 봐야 합니다. 이럴 때 `Content`를 Aggregate Root로 두고, 상태 전이와 평점 부여 같은 규칙을 그 경계 안에서 관리합니다.

여기서 한 걸음 더 나가면 도메인 이벤트와도 자연스럽게 연결됩니다. 예를 들어 `Content`가 `Done` 상태로 전이되면, Aggregate 내부에서는 "완료되지 않은 콘텐츠에는 Rating을 줄 수 없다" 같은 규칙을 먼저 지켜야 합니다. 그리고 그 상태 변경이 정상적으로 끝난 뒤에는 `ContentCompleted` 같은 도메인 이벤트를 발행해, 바깥에서 통계 집계나 추천 갱신 같은 후속 작업이 일어나게 만들 수 있습니다. 즉 Aggregate는 **경계 안의 규칙을 먼저 지키는 단위**이고, 도메인 이벤트는 **그 변경 사실을 경계 밖에 알리는 수단**이라고 볼 수 있습니다.

핵심은 이것입니다.

- Aggregate 밖에서는 내부 객체를 마음대로 바꾸지 않는다
- 상태 변경은 Aggregate Root를 통해서만 일어난다
- 그 경계 안에서 필요한 불변식을 함께 지킨다

비유하자면, 다른 부서의 직원에게 직접 요청하지 않고 반드시 부서장을 통해 요청하는 것과 비슷합니다. Aggregate Root는 그 경계의 대표자이고, 외부는 내부 객체를 직접 조작하지 않고 반드시 그 루트를 통해서만 상태 변경을 요청해야 합니다.

즉 Aggregate는 "객체를 묶는 컨테이너"가 아니라, **규칙을 같이 지켜야 하는 단위**입니다. 그리고 그 규칙을 지킨 뒤에 어떤 일이 일어났는지를 바깥에 알릴 때, 도메인 이벤트와도 자연스럽게 이어집니다.

## Domain Service — 한 객체에 넣기 애매한 규칙

모든 비즈니스 규칙이 Entity나 Value Object 안에 깔끔하게 들어가지는 않습니다.

예를 들어 고객 도메인에서 이런 경우가 있습니다.

- 두 고객 간 포인트를 이전할 때, 보내는 쪽의 잔액과 받는 쪽의 한도를 함께 판단해야 한다
- 고객 등급 산정 시 주문 이력, 포인트, 가입 기간 등 여러 Entity의 정보를 종합해야 한다
- 한 객체의 책임으로 보기 어려운 정책 판단이 있다

이럴 때는 `Domain Service`를 씁니다. 중요한 점은 "Service"라는 이름이 아니라 역할입니다. `Domain Service`는 **도메인 규칙을 담지만, 특정 Entity 하나의 책임으로 귀속시키기 어려운 로직**을 맡습니다.

즉 Service 레이어에 로직을 다 몰아넣는 것과는 다릅니다. DDD의 `Domain Service`는 여전히 도메인 언어로 말해야 하고, 도메인 모델을 보조해야 합니다.

다만 `Domain Service`를 남용하면 주의해야 합니다. 결국 모든 비즈니스 로직이 Service에 몰리면서 Entity와 Value Object가 다시 빈약한 모델로 돌아갈 수 있기 때문입니다. `Domain Service`는 도메인 모델을 대체하는 것이 아니라, 한 객체의 책임으로 넣기 어려운 규칙만 보조적으로 맡아야 합니다.

여기서 `Application Service`와도 구분할 필요가 있습니다. `Domain Service`가 도메인 규칙 자체를 다루는 반면, `Application Service`는 "고객 생성", "포인트 적립", "프로젝트 완료" 같은 유스케이스의 흐름을 조율합니다. 즉 `Application Service`는 도메인 객체와 Repository를 연결해 작업 순서를 관리하지만, 핵심 도메인 규칙 자체는 여전히 Entity, Value Object, `Domain Service`에 남아 있어야 합니다.

## Repository — 저장 기술이 아니라 도메인 관점의 입구

Repository도 자주 오해되는 개념입니다. DDD에서 Repository는 단순한 DB 유틸리티가 아닙니다. **도메인 객체를 저장하고 다시 불러오는 통로**입니다.

중요한 것은 보통 두 가지입니다.

- 인터페이스는 도메인 쪽에 둔다
- 실제 구현은 인프라 쪽에 둔다

왜 이렇게 나눌까요? 그대로 두면 도메인 코드가 Prisma, TypeORM, SQL 같은 인프라 기술에 직접 의존하기 쉽기 때문입니다. 그렇게 되면 가장 오래 살아남아야 할 도메인 레이어가, 가장 자주 바뀌는 하위 레이어에 끌려가게 됩니다. 그래서 DDD에서는 의존성 역전(DIP)을 적용합니다. 도메인이 필요한 저장 계약을 인터페이스로 정의하고, 실제 구현은 인프라 레이어에서 그 계약을 구현합니다. 즉 도메인이 인프라에 의존하는 대신, 인프라가 도메인이 정의한 계약에 맞추도록 만드는 것입니다.

예를 들어 도메인에서는 이렇게 생각합니다.

- `CustomerRepository`가 고객을 저장하고 조회한다
- `ContentRepository`가 콘텐츠 Aggregate를 저장하고 조회한다

하지만 이것이 Prisma인지, TypeORM인지, SQL인지, 메모리 저장소인지는 도메인이 알 필요가 없습니다. Repository는 저장 기술을 감추는 추상화이기도 하지만, 더 본질적으로는 **도메인 모델이 세상과 연결되는 입구**입니다.

물론 이런 분리는 비용도 만듭니다. 도메인 모델, DB 모델, DTO 모델이 분리되면 그 사이를 변환하는 매핑 작업이 필요하기 때문입니다. 이런 작업은 단순하지만 반복적이라 부담이 되기 쉽습니다. 다만 최근에는 이런 종류의 보일러플레이트를 AI가 비교적 잘 보조할 수 있기 때문에, 뒤쪽 장에서는 이 부담과 트레이드오프도 함께 다뤄 보겠습니다.

## Mapper — 서로 다른 모델 사이를 번역하는 역할

Repository 바로 옆에서 자주 함께 등장하는 개념이 `Mapper`입니다. Mapper는 도메인 모델, DB 모델, DTO처럼 **서로 다른 형태의 모델 사이를 변환하는 역할**을 맡습니다. 예를 들어 DB에서 읽어 온 `name: string`, `email: string`, `points: number`를 도메인 안에서는 `Name`, `Email`, `RewardPoints`로 바꾸고, 반대로 도메인 객체를 저장할 때는 다시 DB가 이해할 수 있는 기본형 필드로 풀어 주는 식입니다.

예전에는 이런 매핑 코드가 반복적이고 귀찮게 느껴져서, "그냥 모델을 하나로 통일하면 안 되나?"라는 생각으로 다시 DB 중심 구조로 돌아가기 쉽기도 했습니다. 실제로 많은 사람이 DDD를 어렵게 느끼는 이유 중 하나가 바로 이 변환 비용입니다.

하지만 AI 시대에는 이 지점의 부담이 예전보다 훨씬 줄었습니다. Mapper는 창의적인 비즈니스 규칙보다는, 이미 정해진 도메인 구조를 다른 형태로 옮기는 반복 작업에 가깝습니다. 이런 종류의 코드는 AI가 비교적 안정적으로 초안을 만들기 좋습니다. 그래서 지금은 매핑 코드가 귀찮다는 이유만으로 도메인 모델링을 포기해야 할 이유가 예전보다 작아졌습니다. 오히려 **도메인 규칙은 사람과 도메인 모델에 남기고, 반복적인 변환 작업은 AI의 도움을 받는 구조**가 훨씬 자연스러워졌습니다.

## Domain Event — 분리된 Aggregate를 느슨하게 연결하는 신호

DDD를 조금 더 진행하다 보면, 하나의 Aggregate 안의 규칙만으로는 끝나지 않는 순간이 옵니다. 어떤 Aggregate에서 일어난 중요한 변화가 다른 Aggregate나 다른 기능에도 영향을 주기 때문입니다. 이럴 때 자주 사용하는 것이 `Domain Event`입니다.

Domain Event는 말 그대로 **도메인에서 의미 있는 일이 일어났다는 사실**을 표현하는 이벤트입니다. 예를 들어 `Content` Aggregate에서 소비 완료가 일어났다면 `ContentCompleted` 같은 이벤트를 발행할 수 있습니다. 그러면 통계 Aggregate나 추천 기능은 이 이벤트를 받아 각자 필요한 후속 작업을 수행할 수 있습니다.

이 방식의 장점은, 서로 분리된 Aggregate들이 직접 서로를 호출하지 않고도 협업할 수 있다는 점입니다. 데이터 중심 구조에서는 이런 후속 작업이 보통 Service 레이어에 모입니다. 예를 들어 `contentService.complete()` 안에서 통계 서비스도 호출하고, 추천 서비스도 호출하고, 알림 서비스도 직접 호출하는 식입니다. 처음에는 단순해 보이지만, 시간이 지나면 하나의 Service가 여러 기능을 알고 있어야 하고, 변경이 생길 때마다 연쇄적으로 수정이 퍼지면서 코드가 강하게 결합되기 쉽습니다.

반면 Domain Event를 사용하면 `Content` Aggregate는 "콘텐츠가 완료되었다"는 사실만 알리고, 그 이후 무엇을 할지는 바깥의 다른 모듈이 이벤트를 받아 스스로 결정하게 만들 수 있습니다. 이렇게 하면 Aggregate 사이의 결합은 느슨해지고, 각 모듈은 자기 책임 안에 더 잘 머물 수 있습니다. 즉 도메인 이벤트는 **분리된 Aggregate가 loosely coupled하게 협업하게 만드는 수단**이라고 볼 수 있습니다.

물론 모든 것을 이벤트로 풀어야 한다는 뜻은 아닙니다. 같은 Aggregate 안에서 즉시 함께 지켜야 하는 규칙은 여전히 Aggregate 내부에서 처리해야 합니다. Domain Event는 그 경계 안의 규칙을 지킨 뒤, 그 결과를 경계 밖에 알리고 다른 모듈이 반응하게 만들 때 특히 힘을 발휘합니다.

## OOP와 FP는 대립이 아니라 역할 분담에 가깝다

DDD를 이야기하면 종종 이런 질문이 따라옵니다. "요즘은 객체 지향보다 함수형 프로그래밍이 더 낫다고 하던데, DDD는 여전히 유효한가?" 제 생각은 그렇습니다. DDD의 핵심은 OOP 진영에 서는 것이 아니라, **도메인 규칙이 가장 잘 드러나는 표현을 선택하는 것**입니다.

보통은 이렇게 나눌 수 있습니다.

- 상태와 정체성을 가진 대상은 객체로 모델링한다
- 계산, 변환, 판정 중심 로직은 함수로 표현한다

예를 들어:

- `content.markInProgress()`는 객체로 표현하기 좋습니다
- `calculateRecommendationScore(content, history)`는 함수로 표현하기 좋습니다

즉, OOP와 FP는 경쟁 관계라기보다 **도메인을 더 선명하게 드러내기 위한 서로 다른 도구**입니다.

## DDD가 처음에는 어려운 이유

여기까지 읽으면 한편으로는 이런 생각이 들 수 있습니다. "결국 예전에는 Service 레이어에서 하던 일을, Value Object, Entity, Aggregate, Repository, Mapper 같은 더 많은 개념과 파일로 나누는 것 아닌가?" 어느 정도는 맞는 느낌입니다.

데이터 중심 구조에서는 많은 규칙이 Service 레이어에 모여 있었지만, DDD에서는 그것들이 **주체적인 작은 Building Block**으로 분리됩니다. 그 결과 소스 코드 파일도 더 작은 단위로 나뉘고, 처음에는 오히려 장황하거나 복잡하게 느껴질 수 있습니다. 개념도 늘어나고, 파일도 많아지기 때문에 "이게 정말 더 단순한가?"라는 의문이 드는 것이 자연스럽습니다.

실제로 이 러닝 커브는 DDD를 적용할 때 많은 사람이 겪는 큰 어려움 중 하나입니다. 이런 개념이 아직 손에 익지 않은 상태에서는, Value Object, Entity, Aggregate, Repository, Mapper 같은 분리가 오히려 소스를 더 보기 어렵게 만들 수도 있습니다. 무엇이 어디까지의 책임인지 감이 잡히기 전에는, 파일이 많아진 만큼 복잡도도 같이 늘어난 것처럼 느껴지기 쉽습니다.

하지만 이 분리는 그냥 쪼개기 위한 분리가 아닙니다. **서로 연관되지 않아야 할 것들이 섞이지 않도록, 응집력이 높아지게 분리한 결과**입니다. 처음에는 낯설고 번거롭게 느껴질 수 있지만, 익숙해지고 나면 변경이 생겼을 때 어디를 봐야 하는지가 오히려 더 분명해집니다. 즉 DDD의 어려움은 단순히 파일이 많아서가 아니라, 문제를 바라보는 기준이 Service 중심에서 도메인 객체 중심으로 바뀌기 때문이라고 볼 수 있습니다.

## 여기까지가 이 시리즈를 위한 DDD 기본 어휘

정리하면 이번 글에서 소개한 DDD의 핵심 어휘는 다음과 같습니다.

- **Value Object**: 의미 있는 값을 표현하는 객체
- **Entity**: 식별자와 생명주기를 가진 객체
- **Aggregate**: 규칙을 함께 지켜야 하는 일관성 경계
- **Domain Service**: 특정 객체 하나로 귀속시키기 어려운 도메인 규칙
- **Repository**: 도메인 객체를 저장하고 불러오는 입구
- **Mapper**: 도메인 모델과 DB/DTO 모델 사이를 변환하는 역할
- **Domain Event**: 도메인에서 일어난 중요한 변화를 바깥에 알리는 신호

이제 중요한 것은 용어를 외우는 것이 아닙니다. **요구사항을 읽었을 때 어떤 것이 값이고, 어떤 것이 주체이고, 어떤 규칙이 어디까지 함께 묶여야 하는지 판단할 수 있는가**입니다. 이번 글은 그 판단을 시작하기 위한 최소한의 어휘를 정리한 셈입니다.
