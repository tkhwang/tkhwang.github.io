이 글은 **[SDD+TDD+DDD] 시리즈**의 네 번째 글입니다. <br />

Ch.3에서는 DDD의 기본 개념을 정리했습니다. Value Object, Entity, Aggregate가 각각 무엇인지, 그리고 왜 도메인 규칙을 코드 구조에 맞춰 담아야 하는지 살펴봤습니다. 여기서 바로 구현으로 들어가면 한 가지가 빠집니다. **애초에 어떤 객체를 만들어야 하는지는 어떻게 정하느냐**는 질문입니다.

이번 글에서는 DDD의 기본 개념을 실제 요구사항에 적용해 보겠습니다. 먼저 **무엇을 모델링할 것인지**를 정하고, 어떤 값을 Value Object로 보고 어떤 대상을 Entity로 볼지 판단합니다. 이것이 설계의 시작입니다.

## 설계가 먼저다

DDD 관점으로 설계를 시작하면, 가장 먼저 해야 할 일은 요구사항 안에서 **의미 있는 값과 주체를 구분하는 것**입니다. 어떤 것은 규칙이 붙은 값으로 봐야 하고, 어떤 것은 식별자와 생명주기를 가진 주체로 봐야 합니다. 그 판단이 끝나야 비로소 구현도 자연스러워집니다. 그리고 그다음 단계에서, 설계로 정한 객체의 규칙을 Spec으로 정리하면 SDD가 힘을 발휘합니다.

즉 흐름은 이렇습니다.

1. 요구사항을 읽는다
2. 도메인에서 의미 있는 값과 주체를 찾는다
3. Value Object와 Entity 후보를 나눈다
4. 그다음에 각 객체의 규칙을 Spec으로 정리한다

이번 글에서는 1~3을 다룹니다.

## 요구사항을 다시 읽어 보자

Customer 예제의 요구사항입니다.

- **name**: 비어있을 수 없고, 50자 이하, 앞뒤 공백 제거
- **email**: 유효한 이메일 형식이어야 하고, 대소문자 구분 없이 정규화
- **points**: 0 이상의 정수, 소수점 불가

여기에 Customer 자체를 생각해 보면 보통 이런 사실도 함께 따라옵니다.

- 고객은 고유한 식별자(id)를 가진다
- 고객은 생성되고, 수정되고, 조회되는 생명주기를 가진다

이 시점에서 중요한 것은 데이터 타입이 아니라 **규칙의 위치**입니다. 규칙이 붙는 값은 무엇인지, 시스템 안에서 독립적인 주체로 다뤄야 하는 것은 무엇인지 구분해야 합니다.

## 규칙이 붙는 값부터 찾는다

요구사항을 한 줄씩 읽어 보면, 세 값이 눈에 띕니다.

### Name

이름은 단순한 문자열이 아닙니다.

- 비어 있을 수 없다
- 50자를 초과할 수 없다
- 앞뒤 공백을 정리해야 한다

즉 `name`은 `"문자열 하나"`가 아니라, **생성 규칙과 불변 조건이 붙은 값**입니다.

### Email

이메일도 마찬가지입니다.

- 유효한 형식이어야 한다
- 공백을 제거해야 한다
- 대소문자를 정규화해야 한다

이 역시 단순한 `string`이 아니라, **형식과 정규화 정책을 가진 값**입니다.

### RewardPoints

포인트는 숫자처럼 보이지만, 사실은 규칙이 더 많습니다.

- 0 이상이어야 한다
- 정수여야 한다
- 적립과 차감 같은 행동이 붙는다

즉 `points`는 단순한 `number`가 아니라, **자기 규칙과 행동을 가진 값**입니다.

이 세 값은 모두 같은 방향을 가리킵니다. 시스템 전체에서 반복해서 검증될 가능성이 높고, 의미가 분명하며, 규칙이 값 자체에 붙어 있습니다. 이런 값은 Value Object 후보로 보기 좋습니다.

## 주체가 되는 것은 Entity 후보로 본다

반면 `Customer`는 결이 다릅니다. 고객은 보통:

- 고유한 식별자를 가지고
- 시간에 따라 상태가 바뀌고
- 시스템 안에서 추적되는 대상입니다

이런 대상은 값 자체보다 **정체성(identity)** 과 **생명주기(lifecycle)** 가 중요합니다. 그래서 `Customer`는 `Name`, `Email`, `RewardPoints`와 같은 Value Object가 아니라, Entity 후보로 보는 것이 자연스럽습니다.

정리하면 이렇게 나눌 수 있습니다.

| 후보 | 왜 중요한가 | 분류 |
|---|---|---|
| **Name** | 이름 규칙과 정규화가 붙는다 | Value Object |
| **Email** | 형식 검증과 정규화 정책이 있다 | Value Object |
| **RewardPoints** | 정수/0 이상/적립·차감 규칙이 있다 | Value Object |
| **Customer** | 식별자와 생명주기를 가진다 | Entity |

이 표가 설계의 출발점입니다.

## 잘못 나누면 어떤 문제가 생기는가

이 분류가 왜 중요한지, 반대 사례를 하나 생각해 보겠습니다. 만약 `Name`을 Value Object가 아니라 Entity로 다뤘다면 어떻게 될까요?

Entity는 고유한 식별자와 생명주기를 가집니다. `Name`에 id를 부여하고, DB에 별도 테이블을 만들고, `Customer`가 `Name`을 외래 키로 참조하는 구조가 됩니다. 이름을 수정하면 기존 Name 레코드를 업데이트할 것인지, 새 레코드를 만들 것인지 결정해야 합니다. 같은 이름을 가진 두 고객이 같은 Name Entity를 공유할 것인지도 고민이 됩니다.

하지만 이름은 정체성이 아니라 **값** 입니다. `"홍길동"`이라는 이름 두 개는 그냥 같은 값입니다. 고유한 id를 추적할 이유가 없습니다. 별도 테이블도 불필요한 복잡성만 만듭니다. 이름은 규칙이 붙은 값이지, 생명주기를 가진 주체가 아닙니다.

반대로 `Customer`를 Value Object로 다루면 어떨까요? 고객에게 id가 없으니, 이름과 이메일이 같은 두 고객을 구분할 수 없습니다. 고객의 포인트가 바뀌면 기존 객체를 수정하는 것이 아니라 매번 새 객체를 만들어야 합니다. 시간에 따른 변화를 추적할 방법이 없습니다.

**분류가 어긋나면 불필요한 복잡성이 생기거나, 필요한 기능을 표현할 수 없게 됩니다.** "이 대상은 값인가, 주체인가?"라는 질문이 설계의 첫 번째 판단 기준인 이유입니다.

## 이번 단계에서는 왜 Value Object부터 시작할까

이제 질문이 하나 남습니다. Entity도 있는데, 왜 이번 시리즈에서는 Value Object부터 시작할까요? 이유는 단순합니다. **작고 경계가 분명하기 때문입니다.** Value Object는 도메인 모델에서 가장 작은 설계 단위에 가깝습니다. 작은 단위부터 시작하면 설계도 bottom-up으로 전개하기 쉽습니다. 이름, 이메일, 포인트처럼 규칙이 분명한 값부터 객체로 만들고, 그 위에 Entity와 더 큰 경계를 쌓아 올릴 수 있습니다. 실제로 Entity는 여러 Value Object를 조합해 만들어지는 경우가 많기 때문에, Value Object부터 정리해 두면 이후 설계도 훨씬 수월해집니다.

Value Object는:

- 책임 범위가 좁고
- 규칙이 명확하며
- 외부 의존성 없이 다루기 쉽고
- Spec과 테스트로 옮기기 좋습니다

반면 Entity는 보통 더 많은 결정을 필요로 합니다.

- 식별자는 어떻게 관리할 것인가
- 어떤 상태 전이를 허용할 것인가
- 어떤 Value Object를 포함할 것인가
- 어디까지를 하나의 일관성 경계로 볼 것인가

즉, Value Object는 설계와 SDD를 시작하기에 가장 좋은 단위입니다. 작고 명확하기 때문에, 규칙을 놓치지 않고 Spec으로 정리하기 좋습니다. 저장 관점에서도 이런 값들은 나중에 Entity의 일부로 함께 저장되는 경우가 많아, 실제 구현으로 이어 가기에도 부담이 적습니다.

## 설계의 결과는 "객체 후보 목록"이다

여기까지 오면 아직 코드는 없습니다. Spec도 아직 쓰지 않았습니다. 하지만 중요한 진전이 하나 있습니다. 이제 **무엇을 모델링할지**가 보입니다. 이번 예제의 설계 결과는 다음과 같습니다.

- `Customer`는 Entity 후보다
- `Name`, `Email`, `RewardPoints`는 Value Object 후보다
- 다음 단계에서는 이 Value Object들의 규칙을 Spec으로 정리한다

이 순서가 중요합니다. 설계 없이 바로 Spec으로 들어가면, 왜 이 객체를 다루는지부터 흔들리기 쉽습니다. 먼저 객체 후보를 뽑고, 그다음에 규칙을 정리해야 Spec도 자연스럽게 써집니다.
