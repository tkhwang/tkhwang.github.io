이 글은 **[SDD+TDD+DDD] 시리즈**의 여섯 번째 글입니다. <br />

Ch.5에서는 콘텐츠 북마크와 소비 추적 도메인의 문제를 정의하고, 요구사항을 PRD([`docs/0_PRD.md`](../../docs/0_PRD.md))로 정리했습니다. 이제 남은 질문은 **이 PRD를 DDD 관점에서 어떻게 분해할 것인가**입니다.

## 설계는 Top-Down으로, 구현은 Bottom-up 으로

앞선 글에서는 `Name`, `Email`, `RewardPoints`처럼 작은 값 객체를 예로 들며 DDD의 감각을 익혔습니다. 
그 방식은 입문에는 좋지만, 실제 도메인으로 들어가면 한 단계 위에서 먼저 봐야 합니다.

실무에서는 보통 이런 순서가 더 자연스럽습니다.

1. 이 도메인 안에 어떤 **업무 영역**이 있는가?
2. 이번에 집중할 **Bounded Context**는 어디인가?
3. 그 안에서 어떤 **Aggregate**가 규칙의 중심이 되는가?
4. 그 Aggregate 안에 어떤 **Entity**와 **Value Object**가 들어가는가?

즉 이번 글의 목표는 "요구사항에서 Value Object 몇 개 뽑기"가 아닙니다. **도메인 전체를 보고, 어디서부터 모델링을 시작할지 정하는 것**입니다.

설계는 top-down으로 내려오지만, 구현은 다시 가장 작은 단위부터 시작하는 편이 좋습니다. 그 첫 단위가 Value Object입니다.

이유는 단순합니다.

- 경계가 작다
- 규칙이 분명하다
- 외부 의존성이 적다
- Spec으로 정리하기 쉽다
- 테스트도 빠르게 작성할 수 있다

반면 `Content` Entity나 `Content` Aggregate부터 바로 구현하려고 하면, 상태 전이, 평가 규칙, 저장 방식, 협력 객체까지 한꺼번에 결정해야 합니다. 처음부터 너무 많은 결정을 끌고 들어오게 됩니다.

그래서 이번 시리즈에서는 먼저 top-down으로 전체 구조를 잡고, 실제 구현은 다시 **Value Object부터 bottom-up으로** 쌓아 올리겠습니다. 이 두 방향은 충돌하지 않습니다. 오히려 함께 가야 합니다. **전체 경계는 위에서 보고, 구현은 작은 규칙부터 시작한다**는 뜻입니다.

## 이제 PRD를 DDD 관점에서 분해한다

Ch.5에서 정리한 PRD를 다시 보면, 이미 몇 가지 다른 관심사가 섞여 있습니다.

- 북마크와 상태 전이
- 완료 후 평가
- 통계 계산
- 추천

그래서 곧바로 클래스부터 만들기보다, 먼저 **경계를 나누는 일**부터 시작하는 편이 좋습니다.

## 먼저 Bounded Context를 나눈다

이 요구사항을 작은 도메인 모델로 보면 금방 복잡해집니다. 대신 업무 관심사에 따라 몇 개의 컨텍스트로 나눠 보겠습니다.

### 1. 콘텐츠 북마크/소비 추적 컨텍스트

이 컨텍스트는 가장 핵심입니다.

- 콘텐츠를 등록한다
- 상태를 `Bookmarked → InProgress → Done`으로 전이한다
- 완료한 콘텐츠에 평가를 남긴다
- URL, 태그, 미디어 유형을 관리한다

즉 "콘텐츠 하나가 어떻게 생성되고, 어떻게 소비되고, 어떤 상태를 가지는가"를 책임집니다.

### 2. 통계/인사이트 컨텍스트

이쪽은 소비 기록을 바탕으로 집계를 담당합니다.

- 주간/월간 소비량 집계
- 미디어 유형별 분포 계산
- 태그별 소비 패턴 분석

여기서 중요한 것은 상태 전이 자체가 아니라, **이미 일어난 소비 기록을 어떤 관점으로 읽을 것인가**입니다.

### 3. 추천 컨텍스트

이쪽은 다음에 볼 콘텐츠를 제안하는 역할입니다.

- 완료 이력을 바탕으로 우선순위를 정한다
- 북마크 목록 중 다음 후보를 추천한다
- 선호 미디어 유형이나 태그를 반영한다

즉 추천은 콘텐츠를 생성하거나 상태를 바꾸는 쪽보다, **소비 이력을 해석하고 의사결정을 돕는 쪽**에 가깝습니다.

### 4. 알림/리마인더 컨텍스트

Ch.5에서 너지 알림은 후속 확장 아이디어로 남겨 두었지만, BC 분해 관점에서는 별도 컨텍스트로 식별해 둘 수 있습니다. 이번 시리즈의 본편에서 직접 구현하지는 않더라도, 경계를 미리 나눠 두면 나중에 확장할 때 다른 컨텍스트를 건드리지 않고 붙일 수 있기 때문입니다.

이쪽은 저장해 둔 콘텐츠를 원하는 시간에 다시 꺼내 보도록 돕는 역할입니다.

- 사용자가 알림 시각이나 반복 규칙을 설정한다
- 정해진 시각에 리마인더를 보낸다
- 어떤 콘텐츠를 알릴지 고르거나 우선순위를 정한다

여기서 중요한 것은 콘텐츠 자체의 상태 전이보다, **언제 어떤 조건으로 알림을 보낼 것인가**입니다. 그래서 알림은 `Content`의 생애주기 안에 억지로 넣기보다, 후속 확장으로 별도 컨텍스트로 두는 편이 자연스럽습니다.

이렇게 나눠 보면, 이번 시리즈에서 가장 먼저 붙잡아야 할 곳도 분명해집니다.

## 이번 시리즈는 콘텐츠 북마크/소비 추적 컨텍스트부터 다룬다

통계, 추천, 알림도 중요하지만, 그보다 먼저 **핵심 도메인 데이터가 어떻게 움직이는지**가 정리되어야 합니다. 소비 통계와 추천은 결국 "콘텐츠가 어떤 상태로 존재하고 어떻게 바뀌는가"를 바탕으로 하고, 알림 역시 어떤 콘텐츠를 언제 다시 꺼내 볼지 결정하려면 먼저 콘텐츠의 기본 생애주기가 정리되어 있어야 하기 때문입니다.

그래서 이번 시리즈의 본편은 우선 **콘텐츠 북마크/소비 추적 컨텍스트**에 집중합니다. 이 선택을 하면 Aggregate 후보도 자연스럽게 보입니다.

## Aggregate부터 본다

이 컨텍스트에서 규칙의 중심은 `Content`입니다.
왜 `Content`가 중심일까요?

- 북마크되는 대상이 콘텐츠다
- 소비 상태를 가지는 것도 콘텐츠다
- 완료 여부에 따라 평가 가능 여부가 갈리는 것도 콘텐츠다

즉 여러 규칙이 결국 **콘텐츠 하나의 생애주기**에 묶여 있습니다. 이럴 때는 `Content`를 Aggregate의 중심으로 보는 것이 자연스럽습니다.

### Content Aggregate가 지켜야 하는 규칙

적어도 지금 단계에서는 아래 규칙들이 함께 움직입니다.

- 콘텐츠는 `Bookmarked → InProgress → Done` 순서로 상태가 바뀐다
- 완료되지 않은 콘텐츠에는 `Rating`을 줄 수 없다
- URL, 태그, 미디어 유형은 콘텐츠를 설명하는 값이다

이 규칙들을 한곳에서 다루지 않으면, 어떤 서비스는 완료 전에도 평가를 허용하고, 어떤 서비스는 상태 전이를 건너뛰게 만들 수 있습니다. 그래서 `Content`를 Aggregate로 두고, 그 안에서 일관성을 지키게 하는 편이 좋습니다.

이 시점에서는 저장 구조를 먼저 생각하지 않는 것이 중요합니다. `contents` 테이블 하나를 어떻게 만들지보다, **도메인에서 어떤 규칙이 한 경계 안에서 함께 보장되어야 하는지**를 먼저 보는 것입니다. Aggregate를 이해하기 어렵게 느껴지는 가장 큰 이유 중 하나가 바로 DB 테이블 중심 사고입니다. 여기서는 먼저 **Domain First, Persistence Later**로 접근하겠습니다.

## 그다음에 Entity를 본다

`Content` Aggregate의 중심에는 같은 이름의 `Content` Entity가 있습니다. 이 객체는 식별자와 생명주기를 가집니다.

- 고유한 `contentId`를 가진다
- 생성된다
- 상태가 바뀐다
- 나중에 수정되거나 다시 조회된다

즉 `Content`는 단순한 데이터 묶음이 아니라, 시간이 지나도 같은 대상으로 추적되는 주체입니다. 그래서 `Content`는 Value Object가 아니라 **Entity**입니다.

여기서 중요한 것은 DB row처럼 보는 것이 아니라, **상태 전이와 규칙을 책임지는 주체**로 보는 것입니다. `startConsumption()`, `complete()`, `rate()` 같은 행동이 나중에 이 객체의 중요한 일부가 됩니다.

## 마지막으로 Value Object를 본다

이제 `Content` 안에 어떤 값들이 들어가는지 보겠습니다. 여기서부터는 "그냥 필드"처럼 보이던 것들이 사실은 **규칙이 붙은 값**이라는 점이 보이기 시작합니다.

### ContentUrl

URL은 문자열처럼 보이지만, 실제로는 규칙이 붙습니다.

- 유효한 URL이어야 한다
- 앞뒤 공백 정리가 필요할 수 있다
- 정규화 정책이 필요할 수 있다

즉 `ContentUrl`은 단순한 `string`이 아니라 **검증과 정규화 규칙이 붙은 값**입니다.

### MediaType

미디어 유형도 마찬가지입니다.

- `video`
- `article`
- `podcast`
- `tweet`
- `newsletter`
- `other`

허용 가능한 값의 집합이 정해져 있다면, 이것 역시 의미 있는 값 객체로 다루는 편이 좋습니다.

### Tag

태그는 더 분명합니다.

- 앞뒤 공백을 정리할 수 있다
- 대소문자를 정규화할 수 있다
- 길이 제한이 있을 수 있다

즉 태그는 단순 문자열이 아니라, **정규화 규칙과 제약을 가진 값**입니다.

### ConsumptionStatus

상태 역시 문자열 하나로 넘기기 쉽지만, 사실은 규칙의 중심에 더 가깝습니다.

- `Bookmarked`
- `InProgress`
- `Done`

그리고 중요한 것은 값 자체보다 **전이 규칙**입니다. `Done`에서 다시 `Bookmarked`로 돌아갈 수 있는지, `Bookmarked`에서 바로 `Done`으로 갈 수 있는지 같은 결정이 붙습니다. 이런 값은 단순 enum으로 흘려보내기보다, 필요하다면 행동까지 가진 Value Object로 다루는 편이 좋습니다.

### Rating

평가 역시 숫자 하나로 끝나지 않을 수 있습니다.

- 허용 범위가 있다
- 완료된 콘텐츠에만 줄 수 있다
- 정수만 허용할지 정책이 필요하다

이 중 일부는 `Rating` 자체의 규칙이고, 일부는 `Content` Aggregate와의 협력 규칙입니다. 이런 구분이 보이기 시작하면 설계가 훨씬 명확해집니다.

## 정리하면 이런 구조가 보인다

이번 요구사항을 DDD 관점에서 정리하면 우선 이렇게 볼 수 있습니다.

| 수준 | 후보 | 이유 |
|---|---|---|
| **Bounded Context** | 콘텐츠 북마크/소비 추적 | 상태 전이와 완료 후 평가 같은 핵심 규칙이 모인다 |
| **Bounded Context** | 통계/인사이트 | 소비 기록을 읽고 집계한다 |
| **Bounded Context** | 추천 | 소비 이력을 해석해 다음 후보를 제안한다 |
| **Bounded Context** | 알림/리마인더 | 원하는 시각과 규칙에 맞춰 다시 보기를 유도한다 |
| **Aggregate** | `Content` | 상태 전이와 평가 가능 여부를 함께 지켜야 한다 |
| **Entity** | `Content` | 식별자와 생명주기를 가진 주체다 |
| **Value Object** | `ContentUrl` | URL 검증과 정규화 규칙이 있다 |
| **Value Object** | `MediaType` | 허용 값의 집합이 있다 |
| **Value Object** | `Tag` | 정규화와 길이 제한이 있다 |
| **Value Object** | `ConsumptionStatus` | 상태와 전이 규칙이 있다 |
| **Value Object** | `Rating` | 평가 범위와 관련 규칙이 있다 |

이 표가 중요한 이유는, 이제부터 코드를 어디서 시작해야 할지가 보이기 때문입니다.

### 정리: 이 도메인에서 개념은 이렇게 대응된다

여기까지 오면 `Content`라는 이름이 여러 번 나와 조금 헷갈릴 수 있습니다. 이 도메인에서는 이렇게 대응된다고 보면 됩니다.

| 개념 | 이 도메인에서의 대응 | 역할 | 질문 |
|---|---|---|---|
| **Aggregate Root** | `Content` | 일관성 경계의 진입점, 외부 접근의 창구 | "이 경계 안에서 누가 중심인가?" |
| **Entity** | `Content` | 식별자와 생명주기를 가진 주체 | "이 객체는 시간에 따라 추적되는가?" |
| **Value Object** | `ContentUrl`, `MediaType`, `Tag`, `ConsumptionStatus`, `Rating` | 규칙이 붙은 값 | "이 값은 의미와 제약을 가지는가?" |

Aggregate는 이 표에 없습니다. Aggregate는 별도 클래스가 아니라, Aggregate Root를 중심으로 그 안의 Entity와 Value Object가 함께 규칙을 지키는 **개념적 경계**를 가리킵니다. 지금 Watchable에서는 그 중심이 `Content`입니다. 그래서 `Content`는 식별자와 생명주기를 가진 `Entity`이면서, 동시에 외부에서 그 경계에 접근하는 `Aggregate Root`이기도 합니다. 현재 단계에서는 하위 Entity가 없기 때문에 둘이 같은 객체로 보입니다.

코드로 내려가면 이를 `Entity`, `AggregateRoot`, `Content`의 관계로 표현할 수 있습니다. 다만 `AggregateRoot extends Entity` 같은 베이스 클래스 구조는 여러 표현 방식 중 하나일 뿐입니다. 더 중요한 것은 상속 구조 자체가 아니라, `Content`가 경계의 중심이고 `ContentUrl`, `MediaType`, `Tag`, `ConsumptionStatus`, `Rating` 같은 값들이 그 안에서 함께 규칙을 이룬다는 점입니다.

실제 코드에서는 이를 이런 식으로 표현할 수 있습니다.

```typescript
// 베이스 클래스: Entity는 식별자와 생명주기를 가진다
class Entity {
  id: string;
}

// AggregateRoot는 Entity를 확장한다 (진입점 + 이벤트 발행)
class AggregateRoot extends Entity {
  domainEvents: DomainEvent[];
}

// Content는 AggregateRoot를 상속한다
class Content extends AggregateRoot {
  url: ContentUrl;        // Value Object
  tags: Tag[];            // Value Object
  status: ConsumptionStatus; // Value Object
}
```

즉 `Content`는 `AggregateRoot`를 상속하고, `AggregateRoot`가 이미 `Entity`를 확장하고 있으므로, `Content`는 Entity이면서 동시에 AggregateRoot입니다. 표에서 세 줄로 나뉘어 있던 것이, 코드에서는 하나의 상속 체인(`Entity → AggregateRoot → Content`)으로 표현됩니다.

### 이 구조는 DB에 어떻게 저장될까

도메인에서 `Aggregate`는 개념적 경계이지만, DB에 그대로 `Aggregate`라는 구조가 저장되지는 않습니다. 실제로 저장되는 것은 그 경계의 중심인 `Content`와, 그 안에 포함된 값들입니다.

- `Content`는 보통 `contents` 테이블의 한 row로 저장된다
- `ContentUrl`, `MediaType`, `ConsumptionStatus`, `Rating` 같은 단일 값은 `contents`의 컬럼으로 풀려 저장될 수 있다
- `Tag`처럼 여러 개를 가질 수 있는 값은 별도 테이블(`content_tags`)이나 JSON 컬럼 같은 방식으로 저장할 수 있다

중요한 것은 저장 구조를 먼저 정하는 것이 아니라, **도메인에서 어떤 규칙이 함께 움직이는지 먼저 정한 뒤** 그에 맞는 persistence 구조를 선택하는 것입니다. 즉 `Aggregate`는 DB 테이블 이름이 아니라, 도메인에서 일관성을 관리하는 경계입니다.

## 설계의 결과는 "객체 후보 목록"이다

여기까지 오면 아직 코드도 없고, Spec도 없습니다. 하지만 설계의 결과는 이미 나왔습니다.

- 이번 시리즈의 핵심 컨텍스트는 `콘텐츠 북마크/소비 추적 컨텍스트`다
- 핵심 Aggregate는 `Content`다
- 그 중심 Entity도 `Content`다
- `ContentUrl`, `MediaType`, `Tag`, `ConsumptionStatus`, `Rating`은 Value Object 후보다

즉 이제 "무엇을 모델링할 것인가"가 정해졌습니다. 
다음 단계는 이 후보들 중 작은 것부터 골라, 규칙을 문서로 명확하게 적는 것입니다. 그래서 다음 글에서는 다시 Value Object부터 내려가게 됩니다.
